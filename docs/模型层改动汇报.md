## 1. 基线架构说明（改动前状态）

### 1.1 原始推理管线（`2126377`，darkside，2026-03-09）

原始设计采用自实现的 `SimpleGraph`（仿 LangGraph StateGraph），推理流程如下：

```
用户输入
   ↓
[_node_intent]  ── LLM 调用，专用 prompt 判断意图类型
   ↓
[_route_intent] ── 条件路由
   ├── "irrelevant" → 直接拒绝，返回提示，END
   ├── "knowledge"  → LLM 直接回答，END
   └── "consultation" → [_node_analysis] → [_node_retrieve] → [_node_reason] → 生成报告，END
```

意图分类节点使用独立专用 prompt：

```
- consultation: 具体患者问诊或病例分析
- knowledge:    脑卒中通用知识询问（无具体患者细节）
- irrelevant:   非脑卒中医疗相关
严格区分：如果有患者具体信息，为consultation；如果是一般性问题，为knowledge；否则irrelevant。
```

**关键特点**：每个输入都必须经过 `_node_intent` 这道门，`graph.ainvoke` 一次性运行全部节点，结果收集完毕后再统一分流输出。`run_clinical_reasoning` 是同步生成器，内部用 `asyncio.new_event_loop().run_until_complete()` 包裹异步调用。

## 2. 第一阶段：流式输出重构

### 2.1 端口与 JWT 对齐（`be230f6`，2026-03-16）

**问题**：Python 模型层读取的 JWT 密钥环境变量名（`SECRET_KEY`）与 Java 后端配置的变量名（`AI_API_SHARED_JWT_SECRET`）不一致，导致 `/model/get_result` 一直返回 401。

**改动**：

```python
# 改动前
SECRET_KEY = os.getenv("SECRET_KEY", "your-secret-key-here")

# 改动后
SECRET_KEY = os.getenv("AI_JWT_SECRET") or os.getenv("SECRET_KEY")
if not SECRET_KEY:
    raise RuntimeError("JWT 密钥未配置！请设置环境变量 AI_JWT_SECRET...")
```

**影响**：修复了 Java ↔ Python 认证失败的根本原因。启动时强校验，避免密钥空值悄无声息地运行。

---

### 2.2 结构化错误码 + 优先级队列 + TokenAggregator（`d4ceab7`，2026-03-17）

**问题**：原有 SSE 事件格式散乱（`{"error": ...}`、`{"result": ...}` 等非标准结构），队列无界（推理速度远超消费速度时内存无限增长），thinking 事件逐 token 发送频率过高。

**改动内容**：

新增三个文件：
- `error_codes.py`：结构化错误码，统一格式 `{"type": "error", "code": "...", "message": "...", "content": "..."}`
- `token_aggregator.py`：TokenAggregator 类，将高频 thinking token 聚合为更大批次后再发出
- `test_token_aggregator.py`：单元测试

`main.py` 核心改动：

```python
# 改动前：无界队列
stream_queue = asyncio.Queue()

# 改动后：有界优先级队列，maxsize=50
stream_queue = asyncio.PriorityQueue(maxsize=50)
# 优先级：error(0) > done(1) > meta(2) > result(3) > thinking(4)
```

`qwen_agent.py` 核心改动：

将 `run_clinical_reasoning` 拆分为两层：
- `_run_clinical_reasoning_core`：内部管线，直接 yield 原始事件
- `run_clinical_reasoning`：外层包装，套 TokenAggregator 聚合 thinking 事件后再发出

```python
# thinking 事件聚合策略：
# - 连续 thinking token 合并，元数据（step/title）保留第一个的值
# - 非 thinking 事件出现前先 flush
# - 内部生成器耗尽后最终 flush 兜底
```

**正面影响**：
- SSE 事件格式统一，前端解析更可靠
- 队列背压保护，防止内存溢出
- thinking 事件批量发送，减少前端 DOM 更新频率

**负面影响/注意事项**：
- TokenAggregator 聚合会引入轻微延迟（通常 <100ms），thinking 内容不再实时逐字出现

### 2.3 最终报告流式输出（打字机效果）（`af919bc`，2026-03-17）

**问题**：最终报告收集全部 chunk 后一次性以 `result` 事件发出，用户需等待整个报告生成完毕才能看到内容。

**改动**：

```python
# 改动前：收集后一次性发送
final_chunks = []
async def collect_final():
    async for chunk in self.medical_assistant.stream_final_report(...):
        final_chunks.append(chunk)
loop.run_until_complete(collect_final())
full_report = "".join(final_chunks)
yield {"type": "result", "content": full_report}

# 改动后：逐块 yield chunk 事件
while True:
    item = chunk_bridge.get()
    if item is _SENTINEL:
        break
    yield item  # 每块立即发出
```

**正面影响**：最终报告实现打字机效果，用户第一个字符的等待时间从"整个报告生成时长"缩短为"首个 token 生成时间"。

### 2.4 全链路异步重构（`38557d8`，2026-03-18）

**问题**：`_run_clinical_reasoning_core` 仍是同步生成器，内部通过 `asyncio.new_event_loop().run_until_complete()` 运行异步代码。这导致：
- 无法在同步生成器中直接 `yield` 流式内容
- 向量检索（`_node_retrieve`）阻塞事件循环
- 需要桥接队列（`chunk_bridge`）做同步/异步转换，代码复杂度高

**改动**：

```python
# 改动前：同步生成器 + 内部 new_event_loop
def _run_clinical_reasoning_core(...) -> Generator[dict, None, None]:
    loop = asyncio.new_event_loop()
    result = loop.run_until_complete(self.graph.ainvoke(initial_state))
    # ... 同步处理结果，再 yield

# 改动后：纯异步生成器
async def _run_clinical_reasoning_core(...) -> AsyncGenerator[dict, None]:
    result = await self.graph.ainvoke(initial_state)
    # ... async for chunk in stream: yield chunk
```

`_node_retrieve` 同步改异步：

```python
# 改动前：同步调用，阻塞事件循环
def _node_retrieve(self, state: dict) -> dict:
    evidence = self._run_tool("retrieve_evidence", questions)

# 改动后：放入线程池
async def _node_retrieve(self, state: dict) -> dict:
    loop = asyncio.get_event_loop()
    evidence = await loop.run_in_executor(None, self._run_tool, "retrieve_evidence", questions)
```

**正面影响**：
- 消除了同步/异步桥接的 `chunk_bridge` 队列，代码大幅简化
- 向量检索不再阻塞 FastAPI 事件循环，其他请求可并发处理
- 打字机效果通过原生 `async for` 实现，稳定可靠

### 2.5 knowledge 路径流式化（`d7265f0`，2026-03-20）

**问题**：knowledge 路径（一般性知识问答）使用 `ainvoke` 一次性返回，与 consultation 路径的打字机体验不一致。

**改动**：

```python
# 改动前：一次性返回
knowledge_response = await self.llm_proposer.ainvoke([HumanMessage(...)])
yield {"type": "result", "content": knowledge_response.content}

# 改动后：逐 token 流式
async for chunk in self.llm_proposer.astream([HumanMessage(...)]):
    content = chunk.content if hasattr(chunk, "content") else str(chunk)
    if content:
        yield {"type": "chunk", "content": content}
```

同时将 irrelevant 拒绝消息的事件类型从 `result` 改为 `chunk`，与其他路径保持一致。

**正面影响**：knowledge 路径也具备打字机效果，用户体验统一。

### 2.6 核心性能优化：graph.ainvoke → 手动逐节点执行（`8798fd3`，2026-03-20）

**背景**：`graph.ainvoke` 一次性运行 intent → analysis → retrieve → reason 全部 4 个节点，期间无任何进度反馈。用户提交后，前端 thinking 面板长时间无更新（数十秒空白期）。

**改动核心**：废弃 `graph.ainvoke`，改为手动逐节点执行并在节点之间穿插 thinking 事件。

```python
# 改动前
result = await self.graph.ainvoke(initial_state)
intent_type = result.get("intent_type", "irrelevant")
# 从 result 提取各节点输出

# 改动后
# 立即发出第一个 thinking，消除空白等待期
yield self._emit_thinking("Start", "🔍 正在分析输入...", ...)

# 手动逐节点执行，节点间穿插 thinking
yield self._emit_thinking("Analysis", "🔬 正在分析病例结构...", ...)
analysis_updates = await self._node_analysis(state)

yield self._emit_thinking("Retrieve", "📚 正在检索循证医学证据...", ...)
retrieve_updates = await self._node_retrieve(state)

yield self._emit_thinking("Reason", "🧠 正在临床推理...", ...)
# ...
```

同时将 MCQ 识别与意图分类合并为单次 LLM 调用，节省 1 次串行等待：

```python
# 改动前：两次串行 LLM 调用
classification = await self._classify_multi_mcq_with_llm(case_text)  # 第1次
# graph.ainvoke 内部再调用 _node_intent                               # 第2次

# 改动后：单次合并调用
classification = await self._classify_and_detect_intent(case_text)    # 1次搞定
intent_type = classification.get("intent_type", "irrelevant")
```

**正面影响**：
- 用户提交后立即看到 thinking 事件，感知等待时间大幅缩短
- 减少 1 次 LLM 串行调用，推理速度有所提升

**负面影响（重要）**：

| 问题 | 说明 |
|------|------|
| `_node_intent` 成为死代码 | `self.graph` 对象虽然还在 `__init__` 中 build（含完整的 `add_node / add_conditional_edges`），但 `graph.ainvoke` 从未被调用，`_node_intent` 和 `_route_intent` 永远不会执行 |
| 意图路由从图结构退化为 if/elif | 原本是 StateGraph 中"intent 节点 → 条件边跳转"的一等公民路由，现在变成主函数里的 if/elif 链 |
| MCQ 规则检测绕过了 irrelevant 判断 | `_is_multi_mcq`（正则匹配 `[A-D]\.` ≥2次）新增后被放置在 irrelevant 判断之前，凡是结构上含两个选项格式的输入都被截走，跳过 irrelevant 检查 |
| 合并 prompt 分类精度下降 | 原 `_node_intent` 的专用 prompt 只做一件事；合并后的 `_classify_and_detect_intent` 同时做 MCQ 识别 + 意图分类，prompt 还注明"intent_type 仅在 is_multi_mcq=false 时有意义"，LLM 注意力分散，对 irrelevant 的分类更容易误判 |

**最终效果（当前状态）**：输入与脑卒中无关的问题，系统大概率不拒绝，而是以 knowledge 路径或 consultation 路径作出回答。这是当前最主要的功能回退。

## 4. 第三阶段：证据质量与引用修复

> 以下 4 个 commit 均为 2026-03-31，集中修复了 AI 幻觉引用的根本原因。

### 4.1 修复 AI 文献引用超出文档库范围（`8fdc9e0`）

**问题**：knowledge 路径和用户追问路径（`_stream_user_questions_fast`）缺少 `SystemMessage`，AI 可以自由生成任意 `《》` 括号内的文献名，不受文档库约束，产生幻觉引用。

**改动**：

```python
# 改动前（knowledge 路径）
async for chunk in self.llm_proposer.astream([HumanMessage(content=knowledge_prompt)]):

# 改动后：注入 system_role 约束文献引用
_knowledge_messages = [
    SystemMessage(content=self.reports.system_role),  # 含文档库白名单
    HumanMessage(content=knowledge_prompt),
]
async for chunk in self.llm_proposer.astream(_knowledge_messages):
```

同样修复了用户追问（`_stream_user_questions_answer`）路径。

**正面影响**：knowledge 路径和追问路径的 AI 文献引用现在受文档库白名单约束，幻觉引用大幅减少。

---

### 4.2 修复证据截断导致中间检索维度丢失（`c4ae555`）

**问题根因**：全局截断阈值 `MAX_EVIDENCE_CHARS = 800` 对三个子问题合并后的证据串整体截断，导致：
- 检索了 3 个临床子问题（维度 1、2、3）
- 合并后字符串仅保留前 800 字
- **维度 2 的全部内容被截断丢弃**，AI 实际只能看到维度 1 前段和维度 3 后段
- 每次输出证据字符数恒定为 825（截断边界固定），AI 推理质量不随子问题数量增加而提升

**改动**：

```python
# 改动前
MAX_EVIDENCE_CHARS = 800  # 整体截断，中间维度必然丢失

# 改动后
MAX_EVIDENCE_CHARS = 2000          # 整体上限（3子问题×600 + headers ≈ 1900）
MAX_EVIDENCE_PER_QUESTION = 600    # 每个子问题独立截断，均衡分配
```

**正面影响**：每个检索维度的证据均能被 AI 看到，临床推理覆盖面从"1-2个维度"提升到"3个维度全覆盖"，报告质量显著提升。

---

### 4.3 补全 proposer/critic 的 SystemMessage 注入（`5bc0f8e`）

**问题**：`_parallel_propose_and_critique` 中 proposer、critic、final 三次 `ainvoke` 均未注入 `SystemMessage`，是 AI 幻觉引用的另一个根源（核心推理路径也未受约束）。

**改动**：

```python
# 改动前：所有调用均无 SystemMessage
proposer_task = self.llm_proposer.ainvoke([HumanMessage(content=proposer_prompt)])
critic_task   = self.llm_critic.ainvoke([HumanMessage(content=pre_critic_prompt)])
final_resp    = await self.llm_proposer.ainvoke([HumanMessage(content=final_prompt)])

# 改动后：全部注入 system_role
proposer_task = self.llm_proposer.ainvoke([
    SystemMessage(content=self.reports.system_role),
    HumanMessage(content=proposer_prompt),
])
critic_task = self.llm_critic.ainvoke([
    SystemMessage(content=self.reports.system_role),
    HumanMessage(content=pre_critic_prompt),
])
final_resp = await self.llm_proposer.ainvoke([
    SystemMessage(content=self.reports.system_role),
    HumanMessage(content=final_prompt),
])
```

同时优化了 ThinkingPanel 展示文案：从"证据 N 字符"改为"N 个临床维度，M 个证据片段"，对用户更有信息量。

**正面影响**：核心推理路径（proposer + critic + final）全部受文献引用约束，幻觉引用根因基本消除。

---

### 4.4 critic 提示词重构 + 文献列表兜底注入（`e6a2fbf`）

**改动**：

1. `config/prompts.yaml`：critic 提示词去掉 JSON 格式输出约束，改为结构化自然语言 + 末行风险等级，释放模型推理空间。

2. `config_loader.py`：`update_doc_list` 方法原本用正则替换 `system_role` 中的文献占位符，正则失败时静默回退（文档库约束完全失效）。修改后：正则失败时**在末尾追加强制覆盖语句**，确保任何 `system_role` 格式下文献列表约束均生效。

3. `main.py`：每次请求生成 12 位 hex `req_id`，注入所有日志前缀，`done` 事件携带 `request_id` 字段，实现端到端请求追踪。

**正面影响**：
- critic 评审不再被格式约束束缚，批判质量提升
- 文献列表注入有了兜底保障，彻底杜绝因格式变化导致约束失效的情况
- 生产环境日志可精确关联到具体请求，排障效率提升

## 5. 第四阶段：任务漂移修复

### 5.1 修复检索子问题任务漂移（`fdf1557`，2026-03-31）

**问题根因**：`_unified_analysis` 生成 `clinical_questions`（检索子问题）时，以"临床紧迫性"为排序标准，未区分用户的具体意图。

用户询问"TOAST 分型"时，LLM 仍倾向于生成"溶栓时间窗是否符合？"等治疗类检索子问题（因为治疗类问题临床紧迫性更高）。检索证据随之偏向治疗维度，最终报告跑偏，未能回答用户的实际问题。

**改动**：

双层防护：

**层 1 — Prompt 意图注入**（在生成 `clinical_questions` 前，向 prompt 注入方向约束）：

```python
_DIAGNOSTIC_KW = {"TOAST", "分型", "病因", "定位", "定性", "鉴别", ...}
_TREATMENT_KW  = {"溶栓", "取栓", "抗凝", "降压", "手术", ...}
_PROGNOSIS_KW  = {"预后", "复发", "康复", "二级预防", ...}

if any(kw in case_text for kw in _DIAGNOSTIC_KW):
    intent_hint = "诊断/分型方向：重点生成定位、定性、病因分型（TOAST）、鉴别诊断类问题，不生成溶栓/取栓等治疗操作类问题。"
elif any(kw in case_text for kw in _TREATMENT_KW):
    intent_hint = "治疗决策方向：重点生成治疗方案、禁忌症、时间窗、用药安全性类问题。"
elif any(kw in case_text for kw in _PROGNOSIS_KW):
    intent_hint = "预后/随访方向：..."
else:
    intent_hint = "综合分析方向：按临床优先级生成最需查证的问题。"
```

**层 2 — 生成后过滤**（检测到诊断意图关键词时，对 LLM 生成结果做后处理过滤）：

```python
_DIAGNOSTIC_INTENT_KW = {"TOAST", "分型", "病因", "定位", "定性", ...}
_TREATMENT_QUESTION_KW = {"溶栓", "取栓", "抗凝", "降压", "手术", ...}

if any(kw in case_text for kw in _DIAGNOSTIC_INTENT_KW):
    filtered = [q for q in clinical_questions if not any(kw in q for kw in _TREATMENT_QUESTION_KW)]
    if filtered:
        clinical_questions = filtered  # 过滤掉治疗类子问题
```

**正面影响**：询问"TOAST 分型/病因/鉴别"时，检索子问题会聚焦诊断维度，最终报告与用户意图对齐，不再漂移到治疗方向。

**局限性**：基于关键词匹配，规则覆盖面有限。未包含关键词的语义相似问题（如"帮我分析一下是什么原因导致的"）仍可能漂移。

## 6. 已知遗留问题与副作用

### 6.1 意图路由退化（最高优先级）

**来源**：`8798fd3`（2026-03-20）

`_node_intent`、`_route_intent`、`_build_graph` 中的完整图结构全部成为死代码，`self.graph.ainvoke` 不再被调用。

**当前现象**：输入与脑卒中完全无关的问题（如"帮我写一首诗"、"今天天气怎么样"），系统大概率不拒绝，而是以 knowledge 路径直接作答。

**根因分析**：
1. `_is_multi_mcq` 正则检测在 irrelevant 判断之前，可绕过 irrelevant 拦截
2. 合并 prompt 的意图分类精度低于专用 prompt，LLM 更倾向分类为 knowledge 而非 irrelevant
3. knowledge 路径自身无拒绝能力，prompt 要求"回答脑卒中问题"但 LLM 不会主动拒绝不相关输入

**修复方向**：
- 选项 A（最小改动）：将意图分类恢复为独立专用 LLM 调用，恢复 `_node_intent` 的执行，并将 irrelevant 判断移到 `_is_multi_mcq` 之前
- 选项 B（推荐）：在 knowledge 路径 prompt 中增加显式拒绝指令和 few-shot 示例，同时在入口统一做一次语义级别的医疗相关性过滤

### 6.2 `self.graph` 对象为死代码

`__init__` 中 `self.graph = self._build_graph()` 仍然存在，每次实例化都会执行 `_build_graph()`（含 `add_node`、`add_edge` 等），但 graph 永远不被调用。建议清理或保留待重启。

### 6.3 快速风险评估无审查

`analyze_patient_risk_fast` 绕过文献检索和 critic 评审，纯依赖 LLM 单次输出。若用于高风险临床决策场景，存在误判风险，需在 UI 层做明确的"仅供参考"提示。

### 6.4 任务漂移修复基于关键词规则

`fdf1557` 的修复依赖关键词集合，词表维护成本存在，且对无关键词的同义表达无效。建议后续改为语义向量分类或在意图分类阶段直接输出意图维度。
