## 三、推理架构对比

### 原版架构（手动编排）

```
run_clinical_reasoning()
  └─ _run_clinical_reasoning_core()          ← ~170 行手动循环
       ├─ _classify_and_detect_intent()       ← MCQ + 意图合并判断
       ├─ if MCQ: _run_multi_mcq()            ← MCQ 路径（10 个函数）
       └─ if consultation:
            ├─ _node_analysis()               ← 手动调用
            ├─ _node_retrieve()               ← 手动调用
            ├─ _node_reason()                 ← 手动调用
            └─ stream_final_report()          ← 手动调用
                 └─ yield {"type":"chunk"...} ← 手写 SSE 事件
```

使用手写 `SimpleGraph` + `_CompiledGraph` 模拟 LangGraph，实际上从未调用 `_build_graph`。

### 新版架构（LangGraph StateGraph）

```
run_clinical_reasoning()
  └─ graph.astream_events(version="v2")      ← LangGraph 原生
       └─ _translate_event()                 ← 事件翻译层
            ├─ on_chain_start → node_start
            ├─ on_chain_end   → node_done | token（fallback）
            └─ on_chat_model_stream → token（仅 STREAMING_NODES）

LangGraph 图结构：
  [intent]
     ↓ _route_intent
     ├── "irrelevant"   → [reject]           → END
     ├── "knowledge"    → [knowledge_answer] → END
     └── "consultation" → [analysis]
                              ↓
                          [retrieve]
                              ↓
                            [reason]
                              ↓
                            [report]         → END
```

## 四、SSE 事件格式变更

### 4.1 Python 层输出（新格式）

| type | 触发时机 | 关键字段 |
|------|----------|----------|
| `node_start` | LangGraph 节点开始执行 | `node`、`label` |
| `node_done` | LangGraph 节点执行完毕 | `node`、`summary` |
| `token` | LLM 流式输出每个 token | `content` |
| `done` | 流正常结束（由 `main.py` 组装） | `request_id`、`name`、`all_info` |
| `error` | 任意层级发生错误 | `code`、`message` |

### 4.2 废弃的旧事件类型

| 旧 type | 废弃原因 | 替代方案 |
|---------|----------|----------|
| `thinking` | 语义模糊，字段不统一 | `node_start` |
| `chunk` | 名称不清晰 | `token` |
| `result` | 与 MCQ 路径强绑定 | `token` |
| `meta` | 拆分为 `done` 携带元数据 | `done.all_info` + `done.name` |
| `heartbeat` | 由 SSE 协议层接管 | `sse-starlette ping_interval=15` |

### 4.3 Java 层事件映射（保持 Vue 向后兼容）

Java 层在 `parseModelLine()` 中做事件翻译，Vue 层无需修改：

| Python 新事件 | Java 映射为 | Vue 侧 |
|--------------|------------|--------|
| `token` | `chunk` | 已有处理逻辑 ✅ |
| `node_start` | `thinking` | 已有处理逻辑 ✅ |
| `node_done` | 静默丢弃 | 无需感知 ✅ |
| `done` | `done` | 已有处理逻辑 ✅ |
| `error` | `error` | 已有处理逻辑 ✅ |

| 改动 | 位置 | 说明 |

## 七、部署注意事项

### 服务器执行顺序

```
1. 备份旧向量库
   cp -r /www/wwwroot/Python-backend/chroma_db_unified /backup/chroma_db_unified_bak

2. 删除旧向量库（chromadb 0.5.x → 1.5.5 数据格式不兼容）
   rm -rf /www/wwwroot/Python-backend/chroma_db_unified

3. 切换依赖环境
   pip install -r requirements_new.txt

4. 启动服务（首次启动将自动重建向量库）
   uvicorn main:app --host 0.0.0.0 --port 8000

5. 联调测试三条路径：
   - consultation（四节点完整链路）
   - knowledge（直接流式回答）
   - irrelevant（拒绝响应）
   - /ai/analyze（独立风险评估，不走主管线）
```

### 回滚方案

| 层级 | 回滚方式 |
|------|----------|
| Python 代码 | `git revert`，切回旧 `requirements.txt` 环境 |
| 向量库数据 | 从备份目录还原 `chroma_db_unified_bak/`（**唯一不可通过代码回滚的风险**） |
| Java 代码 | `git revert` |
