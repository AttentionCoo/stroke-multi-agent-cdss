## 一、 项目定位与总体简介

本项目是一个面向**脑卒中（Stroke）临床急救与连续管理场景**的智能医疗辅助决策支持系统（CDSS），旨在通过人工智能技术提升临床辅助决策支持能力。系统以大语言模型为核心，深度融合**检索增强生成（RAG）、双轴矩阵多智能体推理（Multi-Agent System）与多模态医学知识库**，实现了从患者急诊接入、症状/影像输入到辅助分析、医嘱推荐及长期随访的完整闭环。

与传统通用问答系统不同，本项目并非简单依赖模型参数记忆生成结果，而是以权威医学文献与临床指南为知识底座，通过 **LangGraph 状态机驱动的六节点流水线**——**意图识别 → 病例结构化分析 → 证据混合检索 → 多专家并行推理 → 双重校验反思 → 结构化报告生成**——使每一次回答都具备明确证据来源与逻辑依据。系统强调"**证据先行、过程可解释、结果可验证、时效硬管控**"，在保证智能化水平的同时，显著提升了医学场景下的可靠性与安全性。

### 核心设计理念

| 设计原则 | 具体体现 |
|:---|:---|
| **证据先行** | 所有诊疗建议必须基于内置 12 篇权威指南与实时 PubMed 文献检索 |
| **过程透明** | SSE 流式输出完整 Thinking Chain，前端实时渲染推理步骤 |
| **结果合规** | 三重纵深防御：规则引擎硬拦截 + LLM 反思软审查 + 最终安全卫栏 |
| **时效管控** | 急诊绿道模式自动追踪发病时间窗与 DNT 计时器 |

### 目标用户群体

- 👨‍⚕️ **临床医生与急诊团队**：辅助快速查阅脑卒中相关指南、分析急诊病例，在"黄金1小时"内动态追踪发病时间窗与 DNT（入院到溶栓时间），提供精确的诊疗与用药建议。
- 🎓 **医学教学与医学生**：基于真实脱敏病例自动衍生具有"考核陷阱"的数字病例变体，帮助住培医生理解疾病机制与真实临床鉴别诊疗思路。
- 🩺 **患者持续健康管理**：支持个体化健康数据的动态整合与纵向分析，使 AI 能力从单次问答扩展到连续的二级预防与医疗随访过程。

## 二、 系统整体功能简介

本项目围绕真实医疗使用场景，构建了四大核心功能模块，形成了从数据输入、知识调用到智能分析与长期管理的完整闭环。

### （一） 智能问诊与 AI 辅助分析模块（含急诊绿道）

该模块是系统的核心功能入口，主要通过 **SSE（Server-Sent Events）长连接** 实现流式交互：

- **结构化医学推理**：系统在接收到输入后，首先通过 `IntentNode` 进行意图分类（`consultation` / `knowledge` / `irrelevant`），再通过 `AnalysisNode` 提取结构化临床信息（人口学背景、时间轴、神经系统查体、生命体征、影像学闭环、实验室指标），最终结合多智能体推理生成结构化结果。
- **全维临床视角**：输出维度包括"最可能诊断及依据、解剖定位分析、病理机制解释、置信度评估以及需排除的重要诊断"等，高度贴近真实临床思维。系统强制执行**三步逻辑主线**：解剖定位（Localization）→ 病因定性（Qualitative classification）→ 病理机制整合（Mechanism Integration）。
- **时效动态追踪（急诊绿色通道模式）**：当识别到疑似急性缺血性脑卒中（AIS）患者时，系统界面自动启动**发病时间窗倒计时**与 **DNT（入院到溶栓时间）计时器**。结合多智能体根据患者发病时间（如 $<4.5\text{h}$ 溶栓，$<6\text{h}$ 或 $6\sim24\text{h}$ 取栓），实时高亮提示"距溶栓窗口关闭还剩 X 分钟"，确保急救时效。
- **流式交互体验**：系统通过 FastAPI 的 `EventSourceResponse` 与 Java WebFlux 的 `Flux<SseEmitter>` 协同，将 Python Agent 的异步生成器（`async for`）产出持续推送至前端。用户可以实时看到 AI 的推理思考过程（Thinking）与阶段性结果，显著提升交互体验与信任感。

### （二） 患者管理与个体化分析模块（含 EMR 自动生成）

传统 AI 系统多限于单次随机交互，本项目引入了患者电子档案管理机制。

- **长期档案维护**：用于记录和维护患者基本信息、既往病史、用药史及医生备注等关键数据。后端通过 MyBatis-Plus 操作 MySQL 8.0 持久化存储。
- **动态上下文联动**：单次问诊结束后，后台通过 `ConversationSummaryService` 自动评估每轮对话的信息价值并更新至 `all_info` 上下文字段。后续多轮就诊自动结合历史记录进行个体化风险评估，实现真正的连续性健康管理。
- **结构化电子病历（EMR）对接**：智能体结束推理后，自动将问诊和分析结果转化为标准的高质量**结构化出院小结**或**急诊首程病历**，严格支持 **CDA** 或 **FHIR** 国际医疗数据标准，可无缝导出或对接院内 HIS 系统。

### （三） 医学多模态知识学习与文献检索模块

该模块主要用于支持医学学习、科研支持、多模态联合推理与模型知识底座更新。

- **本地与在线双擎**：
  - **本地知识库**：内置 **12 篇**中国脑卒中相关权威指南与共识（如《中国急性缺血性卒中诊治指南2023》《中国重症卒中管理指南2024》《脑血管病防治指南（2024年版）》等），提供在线 PDF 预览（基于 `pdfjs-dist`）与结构化浏览。
  - **在线文献拓扑**：集成 `PubMedService`，通过 NCBI E-Utilities API 一键抓取最新外文高水平文献列表，按证据等级排序（Practice Guideline > Meta-Analysis > RCT > Review > Case Reports）。
- **RAG 强力支撑**：
  - **文档解析管线**：PDF 通过分块策略切分为语义完整的片段（chunk），同时保留原文来源与页码元数据。
  - **AI Batch QA 衍生引擎**：系统精读医疗 PDF 并自动批量衍生提炼高质量 `Q:A` 对（附带原文页码标签），大幅提升急诊场景下的 BM25 关键词检索召回率。采用批次合并策略（每 10 个 chunk 为一批）降低 API 调用次数。
  - **双路混合检索**：ChromaDB 向量语义检索（DashScope `text-embedding-v2` 嵌入）+ BM25 关键词精确检索双路并发，再经 `gte-rerank` 重排模型深度打分。
- **离线语音查房记录**：支持医生在床旁用语音口述患者情况，系统自动通过语音转结构化文本（ASR + NLP）技术，将其转化为标准的文本数据输入给多智能体矩阵。

### （四） 医学教学与科研考核模块

针对系统提及的"医学教学"与"住培考核"场景。

- **全自动医学双盲数字病例生成器**：系统可利用医学大模型，基于真实脱敏病例自动衍生出具有"多维迷惑性陷阱"的模拟病例（例如：故意在历史病史中加入隐蔽的溶栓绝对禁忌症、或微调化验单指标、或修改发病时间窗）。
- **智能考评反馈**：系统用于考核住培医生的鉴别诊断能力与急诊响应速度，并给出 Agent 视角的步骤扣分项与详细考评报告。

> 💡 **功能整体协同说明**：四大模块并非独立存在——智能问诊模块负责实时分析与时效决策，患者管理模块提供动态个体化数据并输出标准病历，医学知识模块提供权威多模态证据来源，教学科研模块则作为系统的应用延伸与能力验证。四者共同构成"**数据—知识—推理—决策—教学**"的完整链路。

| 核心框架 | Vue 3 (Composition API + `<script setup>`) | ^3.5.22 |
| 构建工具 | Vite | ^7.1.7 |
| 状态管理 | Pinia + pinia-plugin-persistedstate | ^3.0.3 |
| 路由 | Vue Router | ^4.5.1 |
| HTTP 客户端 | Fetch API (原生) + 自封装 request.js | — |
| Markdown 渲染 | marked + DOMPurify (XSS 防护) | ^17.0.1 |

| PDF 预览 | pdfjs-dist + vue-pdf-embed | ^3.11.174 |
| CSS 预处理 | Sass (SCSS) Embedded | ^1.98.0 |
| 代码规范 | ESLint 9 + Prettier | latest |

#### 核心功能模块

- **统一工作区（WorkspaceTabs）**：实现智能问诊、患者管理、医学知识学习的 Tab 切换展示，支持拖拽与快捷键操作。
- **流式渲染引擎（ThinkingPanel + ChatWorkspace）**：重点实现了基于 `ReadableStream` + `morphdom` DOM 差异算法的流式数据接收机制。相比传统的 innerHTML 全量替换，morphdom 仅更新变化的 DOM 节点，大幅减少重绘开销，确保长文本医学报告生成的流畅性与光标位置稳定。
- **医学文档预览（PdfPreviewModal）**：基于 `pdfjs-dist` 实现 PDF 在线预览，支持缩放、翻页、文本选择，用于展示内置的 12 篇脑卒中指南原文。
- **多模态上传**：支持医学影像/检验报告图片的上传（Base64 编码传输），配合 `AvatarUpload` 组件实现拖拽与多图预览，前端自动进行图片压缩（`imageCompress.js`）以控制传输体积。
- **参考文献解析（referenceParser.js）**：自动识别输出中的文献引用标记（如《中国急性缺血性卒中诊治指南2023》第X页），并渲染为可点击的锚点链接。

后端基于 **Spring Boot 3.3.13** + **Spring WebFlux** 构建，运行于 **JDK 21** 环境，负责高并发环境下的核心业务流转与医院系统对接。

| JSON 处理 | Jackson (全局 non_null 配置) | — |

#### 核心能力

- **安全与控制体系**：
  - **JWT 双令牌机制**：`Tokeninterceptor` 校验 Access Token，`RefreshTokenInterceptor` 自动刷新，实现无感续期。
  - **分布式限流熔断**：`RedisRateLimiterInterceptor` 基于 Redis 滑动窗口算法实现 IP 级别的请求频率控制，防止恶意刷接口。
  - **Redisson 分布式锁**：控制急诊高并发环境下的患者数据写入冲突，确保数据一致性。
- **SSE 流式中转引擎（AIStreamingServiceImpl）**：
  - 采用响应式编程模型，通过 `WebClient` 对底层 Python FastAPI 服务进行非阻塞的流式调用。
  - 维护 `SSEEventCache`（环形缓冲区）实现断线续传：前端重连时通过 `Last-Event-ID` 请求断点之后的事件，避免数据丢失。
  - 维护 `OnlineUserTracker` 实现在线用户数统计与监控。
- **文件上传服务（OssDocumentService）**：集成阿里云 OSS，支持医学影像与检验报告的安全存储与 CDN 分发。

```python
# main.py 核心入口伪代码
@asynccontextmanager
async def lifespan(app: FastAPI):
    # 启动时初始化资源
    resources["model"] = MedicalAssistant()          # 多智能体编排器
    resources["naming_model"] = NamingModel()         # 命名实体识别

    resources["context_summary"] = ConversationSummaryService()  # 上下文摘要
    resources["vision_service"] = VisionAnalysisService()        # 多模态视觉
    yield
    # 关闭时清理资源

app = FastAPI(title="Stroke CDSS Model Service", lifespan=lifespan)
```

## 四、 核心技术与模型能力

本项目的核心竞争力在于其面向医疗场景深度定制的智能推理与安全拦截能力。

### （一） LangGraph 六节点状态机推理引擎

系统核心推理流程由 `ClinicalGraphBuilder` 基于 LangGraph 的 `StateGraph` 编译生成，定义了完整的**六节点有向状态流转图**：

```
                    ┌─────────────┐
                    │   intent    │ ◄── 入口节点 (意图分类)
                    └──────┬──────┘
                           │ conditional_edges (intent_type)
              ┌────────────┼────────────┐
              ▼            ▼            ▼
        ┌──────────┐ ┌──────────┐ ┌──────────┐
        │ reject   │ │ knowledge│ │ analysis │
        │ (拒绝)   │ │ (知识问答)│ │ (病例分析)│
        └────┬─────┘ └────┬─────┘ └─────┬────┘
             │             │            │ edge
             ▼             ▼            ▼
           END           END      ┌─────────────┐
                                  │  retrieve   │ ◄── 证据混合检索
                                  └──────┬──────┘
                                         │ edge
                                         ▼
                                  ┌─────────────┐
                                  │   reason     │ ◄── 多专家并行推理
                                  └──────┬──────┘
                                         │ edge
                                         ▼
                                  ┌─────────────┐
                                  │  validate   │ ◄── 双重校验
                                  └──────┬──────┘
                                         │ conditional_edges
                              ┌──────────┼──────────┐
                              ▼ pass     ▼ retry    ▼ fail
                        ┌──────────┐ ┌──────┐  ┌──────────┐
                        │  report  │ │reason │  │  report  │
                        │ (生成报告)│ │(重试) │  │(带警告)  │
                        └─────┬────┘ └──▲───┘  └─────┬────┘
                              │         │(最多3次)   │
                              ▼                     ▼
                            END                   END
```

#### 各节点职责详解

| 节点 | 类名 | 输入状态 | 输出状态 | 核心逻辑 |
|:---|:---|:---|:---|:---|
| **① Intent** | `IntentNode` | `case_text` | `intent_type` | 基于 qwen-turbo 快速分类输入为 consultation/knowledge/irrelevant |
| **② Analysis** | `AnalysisNode` | `case_text`, `all_info` | `context`, `clinical_questions`, `complexity` | 结构化提取人口学、时间轴、NIHSS、影像、实验室等信息；生成 3-5 个关键临床子问题 |
| **③ Retrieve** | `RetrieveNode` | `clinical_questions` | `evidence` | 混合检索（ChromaDB 向量 + BM25 关键词）→ gte-rerank 重排 → 证据聚合（上限 2000 字符） |
| **④ Reason** | `ReasonNode` | `context`, `evidence` | `proposal`, `critique`, `*_advice` | 三专家（全科/神经/药师）**并行推理** → LLM 综合生成 Proposal + Critique |
| **⑤ Validate** | `ValidateNode` | `proposal`, `critique` | `validation_passed`, `feedback` | 规则引擎硬匹配禁忌症 + LLM 反思软审查 → 决定 pass/retry/fail |
| **⑥ Report** | `ReportNode` | 全部状态 | `report` | 根据 `report_mode` 选择模板，生成最终结构化临床报告 |

#### 统一状态模型 (ClinicalState)

所有节点共享同一个 TypedDict 状态对象，确保数据在节点间无缝流转：

```python
class ClinicalState(TypedDict):
    case_text: str                  # 原始病例输入
    all_info: str                   # 历史上下文记忆
    report_mode: str                # 报告模式 (emergency/analysis/outpatient/consultation/fast)
    intent_type: str                # 意图分类结果
    context: Dict                   # 结构化临床信息 (ClinicalContext)
    clinical_questions: List[str]   # 检索子问题
    key_risks: List[str]            # 识别到的关键风险
    complexity: str                 # 病例复杂度
    evidence: str                   # 检索到的循证证据
    proposal: str                   # 专家综合提案 (PROPOSAL)
    critique: str                   # 批判性审查意见 (CRITIQUE)
    user_questions: List[str]       # 用户原始问题
    report: str                     # 最终报告
    generalist_advice: str          # 全科医生独立建议
    specialist_advice: str          # 神经专科医生独立建议
    pharmacist_advice: str          # 临床药师独立建议
    validation_passed: bool         # 是否通过校验
    validation_feedback: str        # 校验反馈信息
    reflection_count: int           # 当前反思次数 (最大 3 次)
```

### （二） 检索增强生成（RAG）机制

系统以医学文献与临床指南作为知识底座，实施证据获取前置，构建了完整的 RAG 管线：

#### 1. 文档预处理与向量化

```
12篇 PDF 指南 → load_pdfs_from_dir() → split_documents() (分块, 保留页码元数据)
    ↓
QAGenerator.generate_qa_for_chunks()  ← 批量 (每10 chunk一批) 调用 qwen-turbo
    ↓
生成高质量 Q:A 对 (附原文来源+页码)  → 作为额外检索源增强召回
    ↓
DashScopeEmbeddings (text-embedding-v2) 向量化 → 存入 ChromaDB (HNSW索引持久化)
    ↓
BM25Retriever 从全部文本构建内存倒排索引
```

#### 2. 混合检索与重排

| 阶段 | 技术 | 参数 | 说明 |
|:---|:---|:---|:---|
| **路 A：向量检索** | ChromaDB + DashScope Embeddings | top_k=4 | 语义相似度匹配，擅长理解同义表达 |
| **路 B：关键词检索** | BM25Retriever (langchain-community) | top_k=4 | 精确匹配高频医学术语 (TOAST/NIHSS/溶栓) |
| **融合** | UnifiedSearchEngine.merge_results() | 去重合并 | 双路结果取并集，按来源去重 |
| **重排** | BGEReranker (DashScope gte-rerank) | top_n=5 | 深度语境相关性打分，二次筛选 |
| **截断** | evidence 上限 | 2000字符 | 防止过长上下文导致"迷失在中间" |

#### 3. 检索查询生成策略

`AnalysisNode` 在结构化病例后会生成 3-5 个精准的临床子问题（`max_sub_questions: 3`），覆盖以下维度：

| 问题类别 | 示例关键词 | 检索目的 |
|:---|:---|:---|
| 诊断类 | TOAST, 分型, 定位, 鉴别, 发病机制 | 明确病因与责任血管 |
| 治疗类 | 溶栓, 取栓, 抗凝, 时间窗, 适应证, 禁忌 | 确定治疗方案与风险 |
| 预后类 | 预后, 复发, 康复, 二级预防, 致残率 | 评估长期预后与管理 |

### （四） 医学多智能体矩阵协同推理机制（Multi-Agent System）

为解决传统单模型医疗决策盲点多、风险高的痛点，系统基于 LangGraph 创新设计了"**业务专家轴（纵向） × 决策行为轴（横向）**"的双轴矩阵多智能体协同架构，模拟三甲医院真实临床的"科室会诊"与"三级把关"流程。

#### 1. 双轴协同拓扑架构图

```text
                    【 决策行为轴 (横向 LangGraph 拓扑) 】
                     Proposer             Critic            Integrator
                  (方案生成智能体)     (风险审查智能体)    (整合反思智能体)
                  ┌─────────────────┐ ┌───────────────┐  ┌────────────────┐
全科医生 (GP) ───►│ 初始全科病情评估 │ │ 基础高危筛查  │  │                │
神经专家 (NS) ───►│ TOAST分型/时间窗│ │ 神经禁忌症挖掘│──►│ 最终结构化决策  │
临床药师 (CP) ───►│ 药物联合配伍方案 │ │ 溶栓出血风险  │  │  (合规临床报告) │
                  └─────────────────┘ └───────────────┘  └────────────────┘
                            │                  │                  ▲
                            └──────────────────┴── [校验失败拦截] ─┘
                                                (触发异步自愈反思流, 最多3次)
```

#### 2. 纵向维度：三位医学专家角色

专家角色配置来自 [expert_config.yaml](model/app/config/expert_config.yaml)，支持热加载与动态启停：

| 专家角色 | 优先级 | system_prompt 核心指令 | 推理侧重 |
|:---|:---|:---|:---|
| 🏥 **全科医生 (GP)** | 1 (最高) | "综合患者各项基础概况与生命体征，给出初步分诊、病情稳定性评估及基础维生建议。" | 整体状态审视、慢病风险分级、离线语音查房数据转化 |
| 🧠 **神经专科医生 (NS)** | 2 | "深挖神经系统查体与发病经过，给出具体的定性、定位诊断、及急诊专科处置建议。" | NIHSS 评分、TOAST 分型、CT/MRI 多模态特征、溶栓/取栓时间窗硬管控 |
| 💊 **临床药师 (CP)** | 3 | "审查患者用药史并发症情况，专注于用药禁忌症、药物相互作用以及推荐的剂量范围。" | 检验报告单审查(INR/血小板)、绝对/相对禁忌症、DDI 配伍风险 |

#### 3. 横向维度：Proposer-Critic-Integrator 推理流水线

**ReasonNode** 的执行流程：

```python
async def run(self, state: ClinicalState) -> Dict:
    # Phase 1: Proposer — 三专家并行推理 (asyncio.gather)
    tasks = [
        self._call_expert("全科医生", state),
        self._call_expert("神经专科医生", state),
        self._call_expert("临床药师", state),
    ]
    results = await asyncio.gather(*tasks)
    
    # Phase 2: Critic — LLM 综合生成 PROPOSAL + CRITIQUE
    synthesis_prompt = self.synthesis_config["prompt_template"].format(
        expert_opinions=format_opinions(results)
    )
    response = await self.llm.ainvoke(synthesis_prompt)
    
    proposal = extract_section(response, "PROPOSAL")
    critique = extract_section(response, "CRITIQUE")
    
    return {
        "generalist_advice": results[0],   # 全科医生建议
        "specialist_advice": results[1],   # 神经专家建议
        "pharmacist_advice": results[2],   # 药师建议
        "proposal": proposal,              # 综合提案
        "critique": critique              # 风险批判
    }
```

### （五） 双重校验与反思自愈机制

`ValidateNode` 实现了系统的最后一道安全防线，采用**静态规则引擎 + 动态 LLM 反思**的双重校验策略：

#### 1. 静态规则引擎（硬拦截）

从 [rules_config.yaml](model/app/config/rules_config.yaml) 动态加载禁忌症规则库：

```yaml
contraindication_rules:
  溶栓:                          # 静脉溶栓绝对/相对禁忌症
  - 近期大手术
  - 活动性出血
  - 血小板<100
  - 血压超180/110
  - 头颅CT高密度
  抗凝:                          # 抗凝治疗禁忌症
  - 出血倾向
  - 活动性溃疡
  双抗:                          # 双联抗血小板禁忌症
  - 既往脑出血史

validation_settings:
  max_reflection_count: 3        # 最大反思重试次数
  enable_rule_engine: true       # 启用规则引擎
  enable_llm_reflection: true    # 启用 LLM 反思
```

校验逻辑：对 Proposal 中的治疗方案关键词（溶栓/抗凝/双抗）进行规则匹配，若命中任何禁忌症规则项，直接判定为**未通过**。

#### 2. 动态 LLM 反思（软审查）

当规则引擎通过（或被禁用）后，调用 LLM 进行深层次的医学逻辑校验：
- 检查 Proposal 与 Critique 的一致性
- 验证是否妥善解决了 Critic 提出的所有高风险点
- 评估是否存在隐含的医疗风险

#### 3. 反思自愈循环

```text
ValidateNode 校验结果路由:
  ┌─ "pass"  → generate_report (进入报告生成)
  ├─ "retry" → reason (回到多专家推理节点, reflection_count += 1)
  │              ↑__ 若 reflection_count >= max_reflection_count (3次)
  │              │   则强制降级为 "fail"
  └─ "fail"  → generate_report (强制输出, 附带安全警告标注)
```

这种设计确保了系统在面对复杂边界案例时的**自愈能力**：不会因为一次校验失败就崩溃，而是有机会自我修正，同时也避免了无限循环的风险。

### （六） 五种报告模式与模板系统

系统支持 5 种报告模式，每种模式对应不同的输出结构与适用场景，配置于 [report_templates.yaml](model/app/config/report_templates.yaml)：

| 模式 Key | 名称 | 适用场景 | 输出结构 |
|:---|:---|:---|:---|
| `emergency` | 急诊完整报告 | 急诊/急性卒中，9 段式完整输出 | 患者信息 + 上下文 + 证据 + Proposal + Critique + 修正后最终版 |
| `analysis` | 深度分析报告 | 疑难病例分析，6 段式分析输出 | 侧重鉴别诊断与机制剖析 |
| `outpatient` | 门诊简洁报告 | 门诊快速参考，5 段式简洁输出 | 浓缩核心信息，便于快速查阅 |
| `consultation` | MDT 会诊报告 | 多学科会诊，6 段式系统分析 | 完整的多学科意见汇总 |
| `fast` | 快速回复 | 简单咨询，3 段式快速输出 | 极简回答，追求低延迟 |

#### 报告生成安全卫栏

所有报告模板均注入统一的 `system_role` 系统提示词（约 380 行），包含以下强制性约束：

- **文献引用规范**：仅允许引用内置 12 篇指南（精确匹配名称），严禁引用库外文献
- **三步逻辑主线强制**：Localization → Qualitative → Mechanism Integration
- **机制闭环要求**：所有判断必须形成"统一病理机制解释"
- **风险权重原则**：出血风险 / 再灌注风险 / 血栓扩展风险 / 呼吸循环失代偿风险 必须排序
- **青年卒中触发条件**：年龄 <50 岁或无传统危险因素时，自动触发反常栓塞/动脉夹层/高凝状态排查
- **安全卫栏五条**：禁止确诊语气 / 禁止具体剂量 / 信息缺失标注"不足" / 风险不明优先保守 / 必须明确不确定性

### （七） 三级 LLM 矩阵与算力分级策略

系统针对不同推理节点的精度与速度需求，部署了三级模型矩阵，实现算力与成本的精细化管理：

| 模型层级 | 模型实例 | 部署节点 | 设计考量 |
| --- | --- | --- | --- |
| **旗舰级** | qwen-max | Proposer（方案生成）、Integrator（综合决策）、3 专家并行推理 | 高参数模型确保临床推理深度与多步反思质量 |
| **均衡级** | qwen-plus | Critic（风险审查）、上下文摘要、Evidence Synthesis（证据综合） | 兼顾推理精度与响应速度，控制审查环节延迟 |
| **极速级** | qwen-turbo | Intent（意图识别）、Knowledge（知识问答）、QA Generator（QA 衍生）、QuickAnalyze（快速分析）、HealthRisk（风险评估）、Summary（摘要）、Naming（命名） | 极低延迟响应，支撑高并发轻量级任务 |

## 五、 创新点与作品亮点

- 🏆 **(一) 证据驱动的医学推理范式创新**：打破传统大模型"先生成再解释"导致的严重医疗幻觉，将证据获取前置。模型在生成答案前必须依赖外部多模态医学知识，实现了从"模型主导"向"证据主导"的范式跨越。内置 12 篇权威指南 + PubMed 实时检索 + 强制文献溯源（名称+页码），使每一句建议都有据可查。

- 🛡️ **(二) 多智能体协同的安全拦截机制**：首创"生成—审查—反思"双轴矩阵架构，Critic 结合多模态检验单识别绝对禁忌症，扮演严苛的上级把关医师。配合 **ValidateNode** 的双重校验（规则引擎硬匹配 + LLM 反思软审查）与**最多 3 次异步自愈反思循环**，在后台实现"方案提出 → 挑刺风险 → 拦截修正"的安全闭环。规则库支持 YAML 热加载，新增禁忌症无需改代码。

- ⏱️ **(三) 具有时效动态硬管控的绿道路径设计**：将"时间就是大脑"的临床急救理念工程化，引入动态时间窗倒计时与 DNT 追踪。系统能根据时间滴答（Tick）动态调整 Agent 推理决策的优先级，体现了对急诊脑卒中场景的深度定制。

- 🔄 **(四) LangGraph 状态机驱动的六节点流水线**：区别于传统的线性 if-else 代码编排，采用 LangGraph 的声明式 StateGraph 定义推理拓扑。状态机的条件边路由（conditional_edges）天然支持意图分流（consultation/knowledge/irrelevant）与校验结果分支（pass/retry/fail），使复杂的医疗业务逻辑变得可视化、可调试、可扩展。新增节点只需添加 node + edge，无需改动现有逻辑。

- 🌊 **(五) 全链路端到端流式架构**：从 Python async generator → FastAPI EventSourceResponse → Java WebClient Flux → Vue3 ReadableStream + morphdom DOM 差异更新的完整流式链路。SSE 断线续传（Last-Event-ID + 环形缓冲区缓存）确保网络抖动时不丢失数据。Token 聚合机制对高频小片段输出进行合并，减少前端 DOM 更新频率。

- 📚 **(六) AI Batch QA 衍生增强 RAG 召回**：创新性地利用 LLM 将长文档 chunks 批量转化为高质量的 Q:A 对（每批 10 个 chunk 合并处理），作为 BM25 检索的额外语料源。这一策略大幅提升了急诊场景下对罕见医学术语和特定数值阈值（如"血压超180/110"）的检索召回率。

## 七、 百度千帆大模型平台能力集成方案

针对本项目高并发、低延迟、强逻辑反思以及多模态的需求，百度千帆大模型平台可提供**全栈式的核心能力支撑**。以下从模型底座、算力调度、RAG 组件、多模态网关与安全合规五个维度，阐述当前系统架构与千帆平台的能力映射与集成路径。

### 1. 异构模型底座与专有域微调 (Model Base & SFT)

多智能体矩阵对专科知识深度与反思逻辑要求极高，千帆平台的模型底座能力可为核心推理节点提供强力支撑。

- **高参数旗舰底座**：将 **qwen-max** 节点（当前用于 Proposer、Integrator 及三专家并行推理）迁移至千帆平台的 **ERNIE 4.0 / ERNIE 4.0 Turbo** 旗舰模型，利用其行业领先的中文理解能力与逻辑推理性能，进一步提升医疗风险审查的严苛度与综合决策的可靠性。
- **千帆 SFT（监督微调）平台**：利用千帆提供的 Post-training 工具链，导入系统内置的国内脑卒中权威指南（12 篇 PDF）及脱敏临床病例，对大模型进行**专科定向微调（SFT）**，训练出专属于"神经专科 Agent"和"临床药师 Agent"的专有域模型，提升 TOAST 分型和禁忌症识别的绝对准确率。微调后的模型可通过千帆 Model Registry 管理与版本控制。

**当前架构映射**：

| 当前模型 | 千帆平台对应 | 当前部署节点 | 集成方式 |
| --- | --- | --- | --- |
| qwen-max（旗舰级） | ERNIE 4.0 / ERNIE 4.0 Turbo | Proposer、Integrator、3 专家并行推理 | 替换 ChatOpenAI 的 model 参数 |
| qwen-plus（均衡级） | ERNIE 3.5 / ERNIE Speed | Critic、上下文摘要、Evidence Synthesis | 替换 ChatOpenAI 的 model 参数 |
| qwen-turbo（极速级） | ERNIE Lite / ERNIE Speed | Intent、Knowledge、QA Generator、Naming | 替换 ChatOpenAI 的 model 参数 |

> **迁移优势**：由于系统已采用 LangChain 的 `ChatOpenAI` 抽象层（OpenAI 兼容协议），且千帆平台同样提供 OpenAI 兼容 API，迁移仅需修改 `base_url`（指向千帆 endpoint）、`api_key`（千帆 API Key）及 `model` 名称，**无需改动任何业务逻辑代码**。

### 2. 算力分级与极致性价比控制 (MaaS 成本优化)

针对系统的"极速通道与安全通道双路径设计"以及 Agent 矩阵的并发调用特性，千帆平台提供统一的模型路由与弹性计费体系。

- **多尺寸模型混用方案**：
  - **日常问答/极速通道/Intent 节点**：调用千帆平台上的 **ERNIE Speed** 或 **ERNIE Lite**。这些模型推理速度极快（首 token 延迟 <500ms），Token 成本约为旗舰模型的 1/10~1/20，适合大规模并行处理轻量任务。
  - **风险审查 (Critic) / 综合决策 (Integrator)**：升维调用 **ERNIE 4.0** 旗舰模型，用高参数模型的强逻辑确保医疗安全。
  - **千帆 Router 智能路由**：可配置自动路由规则，根据输入复杂度自动选择模型级别，进一步优化成本。
- **弹性扩缩容（Serverless 推理）**：千帆平台的 Serverless 推理架构支持按需弹性扩缩，完美匹配急诊高峰期（如冬季卒中高发季、夜间急诊波峰）的突发流量需求，无需预先配置固定算力资源，闲时自动缩容至零节省成本。
- **TPM/RPM 并发配额管理**：千帆提供细粒度的 Tokens Per Minute (TPM) 和 Requests Per Minute (RPM) 配额控制，可与后端 RedisRateLimiter 形成双层限流保障。

### 3. 企业级 RAG 加速与高并发检索组件 (AppBuilder / Reranker)

当前系统的混合检索在面对全量 PubMed 和海量指南时，自建 ChromaDB + BM25 的工程维护成本较高。千帆平台可提供开箱即用的 RAG 基础设施，显著降低运维负担。

- **千帆 AppBuilder / 知识库组件**：
  - 直接调用其内置的**高性能向量检索**与**混合路由策略**（语义检索 + 关键词检索），替代当前自建的 `HybridRetriever`（ChromaDB + BM25Retriever）方案。
  - 内置文档解析引擎（支持 PDF/Word/Excel/图片 OCR），可替代当前的 `load_pdfs_from_dir()` + `split_documents()` 自建管线。
  - 自动分块策略优化（基于语义边界而非固定长度），提升检索精度。
  - 企业级高可用保障（多副本部署、自动故障转移），省去底层繁琐的索引维护与调优工作。
- **千帆 Reranker 模型**：
  - 替换当前使用的 `gte-rerank`（通过 DashScope TextReRank 调用），切换至千帆平台经过大规模中文医疗语料预训练的 **Baidu Reranker** 模型。
  - 在中文医学场景下具备更精准的语义相关性计算能力，尤其对中文医学术语的同义词扩展（如"脑梗死"="脑梗塞"="缺血性卒中"）和语境理解有显著优势。
  - 支持批量重排 API，可一次对多个 query-doc pair 打分，降低网络往返延迟。

**当前架构映射**：

| 当前组件 | 当前实现 | 千帆平台对应 | 能力提升 |
| --- | --- | --- | --- |
| ChromaDB + BM25Retriever（自建混合检索） | DashScope Embeddings + langchain-community BM25 | 千帆 AppBuilder 知识库 | 免运维、企业级 HA、内置混合路由、OCR 解析 |
| gte-rerank（通用语义重排） | DashScope TextReRank (gte-rerank) | 千帆 Baidu Reranker | 中文医疗语料预训练、更精准的医学语义对齐 |
| QAGenerator（QA 批量衍生） | qwen-turbo 批量调用 | 千帆 ERNIE Speed 批量调用 | 更低成本、更高吞吐 |
| PDF 文档解析 | PyPDF2/pdfplumber 自建 | 千帆 AppBuilder 文档解析 | 开箱即用、格式兼容更好 |

### 4. 多模态视觉与语音网关能力 (Multimodal API)

当前系统通过阿里云 DashScope 的 `MultiModalConversation` API 调用 **Qwen-VL-Max** 实现影像与报告单的视觉理解。千帆平台提供对等的原生多模态能力，可作为**双路冗余**或**主路替换**。

- **文心一言多模态视觉接口（ERINE-VL）**：
  - 提供图像语义理解能力，用于解析前端上传的**急诊化验单图片**（血常规、凝血功能、生化全套）、**心电图图表**及**医学影像报告截图**。
  - 自动将图像中的数值指标（如血小板 85×10⁹/L、INR 2.3）转化为高度结构化的 Prompt 文本或直接输出 JSON 格式的结构化指标，无缝喂给 LangGraph 拓扑网络的 ValidateNode 规则引擎。
  - 支持多图联合推理（如同时上传 CT 片 + 化验单），模型可交叉关联分析。
- **语音识别与合成（语音 API）**：
  - 千帆语音识别 API 可替代当前 ASR 方案（若已有集成），实现床旁语音查房记录的实时转写。
  - 支持医疗领域专属声学模型，对医学术语（如"基底节区梗死"、"大脑中动脉闭塞"）的识别准确率显著优于通用模型。
  - 语音合成（TTS）可用于将生成的临床报告朗读出来，方便医生在移动场景下听取摘要。

**当前架构映射**：

| 当前组件 | 当前实现 | 千帆平台对应 | 集成方式 |
| --- | --- | --- | --- |
| Qwen-VL-Max（视觉理解） | DashScope MultiModalConversation.call(model="qwen-vl-max") | 文心一言 ERINE-VL Vision API | 替换 `_run_sync_stream()` 中的 API 调用 |
| ASR（语音转写，规划中） | 待集成 | 千帆语音识别 API | 新增语音查房模块 |

**双路冗余设计建议**：可将千帆 ERINE-VL 作为主路，Qwen-VL-Max 作为备路。当主路 API 异常（超时/限流）时自动 fallback 至备路，提升系统的容错能力与服务可用性。

### 5. 医疗合规与内容安全网关 (Safety Guardrail)

医疗行业对"幻觉"和"错误医嘱"零容忍，千帆平台的安全能力可作为系统的最后一道"硬件防火墙"，与现有的双重校验机制形成**三重纵深防御体系**。

- **千帆大模型安全自护（Guardrails / 内容审核 API）**：
  - 在系统的 Integrator / ReportNode 输出最终结构化临床报告之前，调用千帆的内容安全审核 API 作为一个**独立的第三层安全屏障**。
  - 执行敏感词过滤、医疗违规内容检测（如错误剂量、未经证实的疗法推荐）、合规性检查，防止模型输出任何违反医疗伦理或带有错误偏导向的言论。
  - 可配置自定义审核规则（如"禁止出现具体药物剂量"、"必须包含免责声明"）。
- **Prompt 防注入检测**：千帆 Guardrails 提供 Prompt Injection 检测能力，防止恶意用户通过构造特殊输入绕过系统提示词约束（如在 case_text 中嵌入"忽略以上所有指令"）。
- **与现有双重校验的协同架构**：

```text
Integrator / ReportNode 输出
    │
    ▼
① ValidateNode 规则引擎硬拦截（禁忌症关键词匹配: 活动性出血/血小板<100/...）
    │ 通过
    ▼
② ValidateNode LLM 反思软审查（深层医学逻辑校验: Proposal vs Critique 一致性）
    │ 通过
    ▼
③ 千帆 Guardrails 安全网关（合规性终审: 敏感词/医疗伦理/Prompt注入检测）
    │ 通过
    ▼
④ 最终临床报告输出 (附安全卫栏免责声明)
```

这种**规则引擎 → LLM 反思 → 平台安全网关**的三重纵深防御，确保了医疗输出的绝对安全。即使前两层因极端边界案例未能拦截，千帆 Guardrails 作为独立的第三方审计仍能兜底。

## 八、 千帆平台集成实施路径

基于上述能力映射，系统向百度千帆平台的迁移可按以下**三阶段渐进式**实施，确保业务连续性与平滑过渡：

### 阶段一：模型底座无缝替换（低风险，预计 1-2 天）

**目标**：将当前通过 DashScope OpenAI 兼容接口调用的 Qwen 系列模型，逐步替换为千帆平台的 ERNIE 系列模型。

**实施步骤**：
1. 在千帆平台创建应用，获取 `API Key` 与 `Endpoint URL`
2. 修改 `model/.env` 文件：
   ```env
   # 旧配置
   OPENAI_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1
   OPENAI_API_KEY=sk-dashscope-xxx
   
   # 新配置（千帆）
   OPENAI_BASE_URL=https://qianfan.baidubce.com/v2
   OPENAI_API_KEY=your-qianfan-api-key
   ```
3. 修改 `config/prompts.yaml` 中的模型映射表（如有硬编码模型名）
4. 逐个节点验证：先替换 qwen-turbo 节点（Intent/Knowledge）→ 再替换 qwen-plus 节点（Critic）→ 最后替换 qwen-max 节点（Proposer/Expert/Report）

**风险控制**：由于系统已采用 LangChain 的 `ChatOpenAI` 抽象层，且千帆平台完全兼容 OpenAI API 协议，**无需改动任何业务逻辑代码**，仅需修改配置文件。可保留 DashScope 作为 fallback 备路。

### 阶段二：RAG 基础设施升级（中风险，预计 3-5 天）

**目标**：将自建 RAG 管线迁移至千帆 AppBuilder 知识库组件，同时将 Reranker 切换至千帆原生模型。

**实施步骤**：
1. 在千帆 AppBuilder 创建知识库应用，上传 12 篇指南 PDF
2. 配置知识库的解析策略（分块大小、重叠长度、元数据字段）
3. 修改 `retrievers.py` 中的 `UnifiedSearchEngine`，将 ChromaDB + BM25 调用替换为千帆知识库 API
4. 修改 `retrievers.py` 中的 `BGEReranker`，将 DashScope gte-rerank 调用替换为千帆 Reranker API
5. 运行 RAG 评测套件 (`tests/test_rag.py`) 对比迁移前后的召回率与准确率

**收益**：此阶段完成后，可移除 ChromaDB 本地依赖、DashScope Embedding 依赖、PyPDF2/pdfplumber 依赖，显著简化运维栈。

### 阶段三：安全与多模态能力增强（增量集成，预计 3-5 天）

**目标**：在现有双重校验机制基础上，接入千帆 Guardrails 安全网关作为第三层防御；将多模态视觉能力扩展为千帆 API 双路冗余。

**实施步骤**：
1. **Guardrails 接入**：在 `ReportNode.run()` 返回前增加千帆内容审核 API 调用，审核不通过则附加警告标记而非阻断（避免过度拦截影响体验）
2. **多模态双路冗余**：修改 `VisionAnalysisService._run_sync_stream()`，优先调用千帆 ERINE-VL，失败时 fallback 至 Qwen-VL-Max
3. **语音查房模块（可选）**：若需新增语音功能，集成千帆语音识别 API，在 `main.py` 新增 `/voice/dictate` 端点
4. **端到端回归测试**：运行完整测试套件 (`tests/test_analyze_api.py`, `test_quick_analyze.py`) 验证全链路正确性
