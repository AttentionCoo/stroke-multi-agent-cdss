# 🧠 多智能体深度检索脑卒中临床辅助决策支持系统 (CDSS)

## 一、 项目定位与总体简介

本项目是一个面向**脑卒中（Stroke）临床急救与连续管理场景**的智能医疗辅助决策支持系统（CDSS），旨在通过人工智能技术提升临床辅助决策支持能力。系统以大语言模型为核心，深度融合**检索增强生成（RAG）、双轴矩阵多智能体推理（Multi-Agent System）与多模态医学知识库**，实现了从患者急诊接入、症状/影像输入到辅助分析、医嘱推荐及长期随访的完整闭环。

与传统通用问答系统不同，本项目并非简单依赖模型参数记忆生成结果，而是以权威医学文献与临床指南为知识底座，通过"**极速/安全双通道分流—多模态动态检索—多智能体交叉重排—自愈式反思推理—自动化评估**"的结构化流程，使每一次回答都具备明确证据来源与逻辑依据。系统强调"**证据先行、过程可解释、结果可验证、时效硬管控**"，在保证智能化水平的同时，显著提升了医学场景下的可靠性与安全性。

在应用定位上，本项目面向以下群体，并兼顾多维度的医疗需求：

- 👨‍⚕️ **临床医生与急诊团队**：辅助医生快速查阅脑卒中相关指南、分析急诊病例，在"黄金1小时"内动态追踪发病时间窗与 DNT（入院到溶栓时间），提供精确的诊疗与用药建议。
- 🎓 **医学教学与医学生**：基于真实脱敏病例自动衍生具有"考核陷阱"的数字病例变体，帮助住培医生理解疾病机制与真实临床鉴别诊疗思路。
- 🩺 **患者持续健康管理**：支持个体化健康数据的动态整合与纵向分析，使 AI 能力从单次问答扩展到连续的二级预防与医疗随访过程。

本项目不仅是一个 AI 模型应用，更是一个融合前端交互、后端反应式流处理与分布式模型推理服务的完整系统，具备高并发、高扩展性与实际应用潜力，为构建真实的院内 CDSS 提供了可行的技术路径。

## 二、 系统整体功能简介

本项目围绕真实医疗使用场景，构建了四大核心功能模块，形成了从数据输入、知识调用到智能分析与长期管理的完整闭环。

### （一） 智能问诊与 AI 辅助分析模块（含急诊绿道）

该模块是系统的核心功能入口，主要用于处理用户输入的症状描述、病史信息或临床问题。

- **结构化医学推理**：系统在接收到输入后，首先进行医学问题理解与任务拆解，再通过检索获取相关证据，最终结合多智能体推理生成结构化结果。
- **全维临床视角**：输出维度包括"最可能诊断及依据、解剖定位分析、病理机制解释、置信度评估以及需排除的重要诊断"等，高度贴近真实临床思维。
- **时效动态追踪（急诊绿色通道模式）**：当识别到疑似急性缺血性脑卒中（AIS）患者时，系统界面自动启动**发病时间窗倒计时**与 **DNT（入院到溶栓时间）计时器**。结合多智能体根据患者发病时间（如 $<4.5\text{h}$ 溶栓，$<6\text{h}$ 或 $6\sim24\text{h}$ 取栓），实时高亮提示"距溶栓窗口关闭还剩 X 分钟"，确保急救时效。
- **流式交互体验**：系统支持流式输出，用户可以实时看到 AI 的推理思考过程（Thinking）与阶段性结果，显著提升交互体验与信任感。

### （二） 患者管理与个体化分析模块（含 EMR 自动生成）

传统 AI 系统多限于单次随机交互，本项目引入了患者电子档案管理机制。

- **长期档案维护**：用于记录和维护患者基本信息、既往病史、用药史及医生备注等关键数据。
- **动态上下文联动**：用户补充新的症状后，系统会自动结合历史记录与当前输入进行综合分析，实现个体化风险评估与辅助诊断。这使 AI 能够真正参与到患者的长期健康管理与动态评估中。
- **结构化电子病历（EMR）对接**：智能体结束推理后，自动将问诊和分析结果转化为标准的高质量**结构化出院小结**或**急诊首程病历**，严格支持 **CDA** 或 **FHIR** 国际医疗数据标准，可无缝导出或对接院内 HIS 系统。

### （三） 医学多模态知识学习与文献检索模块

该模块主要用于支持医学学习、科研支持、多模态联合推理与模型知识底座更新。

- **本地与在线双擎**：内置多篇国内权威脑卒中指南并提供 PubMed 在线接口，支持结构化浏览与快速阅读。
- **RAG 强力支撑**：这些专业文献不仅用于阅读，更是系统检索增强（RAG）的核心知识来源。系统通过文档解析、语义分块和向量化索引，使 AI 输出严格建立在真实医学资料基础之上。
- **离线语音查房记录**：支持医生在床旁用语音口述患者情况，系统自动通过语音转结构化文本（ASR + NLP）技术，将其转化为标准的文本数据输入给多智能体矩阵。

### （四） 医学教学与科研考核模块

针对系统提及的"医学教学"与"住培考核"场景。

- **全自动医学双盲数字病例生成器**：系统可利用医学大模型，基于真实脱敏病例自动衍生出具有"多维迷惑性陷阱"的模拟病例（例如：故意在历史病史中加入隐蔽的溶栓绝对禁忌症、或微调化验单指标、或修改发病时间窗）。
- **智能考评反馈**：系统用于考核住培医生的鉴别诊断能力与急诊响应速度，并给出 Agent 视角的步骤扣分项与详细考评报告。

> 💡 **功能整体协同说明**：四大模块并非独立存在——智能问诊模块负责实时分析与时效决策，患者管理模块提供动态个体化数据并输出标准病历，医学知识模块提供权威多模态证据来源，教学科研模块则作为系统的应用延伸与能力验证。四者共同构成"**数据—知识—推理—决策—教学**"的完整链路。

## 三、 系统架构与前后端设计

本项目采用前后端分离与模型服务独立部署的三层架构，将用户交互、业务逻辑与 AI 推理能力进行深度解耦。

```text
┌───────────────────────────────────────────────────────────────────────────┐
│                          前端交互层 (Vue 3 / SSE)                          │
│ ┌──────────────────┐  ┌────────────────────┐  ┌─────────────────────────┐ │
│ │ 绿道时效倒计时面板 │  │ 多模态影像/报告预览 │  │ 思考过程与病历流式渲染   │ │
│ └─────────┬────────┘  └──────────┬─────────┘  └────────────▲────────────┘ │
└───────────┼──────────────────────┼─────────────────────────┼──────────────┘
            │ (RESTful / WebSocket)│                         │ (SSE 流式响应)
┌───────────▼──────────────────────▼─────────────────────────┴──────────────┐
│                    后端服务层 (Spring Boot / WebFlux)                     │
│ ┌──────────────────┐  ┌────────────────────┐  ┌─────────────────────────┐ │
│ │  安全认证 (JWT)   │  │  分布式锁并发控制   │  │ WebClient 响应式转发引擎 │ │
│ └─────────┬────────┘  └──────────┬─────────┘  └────────────▲────────────┘ │
└───────────┼──────────────────────┼─────────────────────────┼──────────────┘
            │                      │                         │ (Async Generator)
┌───────────▼──────────────────────▼─────────────────────────┴──────────────┐
│                    模型推理服务层 (FastAPI / Python)                       │
│  ┌─────────────────────────────────────────────────────────────────────┐  │
│  │ 混合检索 (ChromaDB+BM25) ──► Reranker 重排 ──► 证据精准溯源        │  │
│  ├─────────────────────────────────────────────────────────────────────┤  │
│  │ 多模态视觉网关 (Qwen-VL-Max): 解析 ASPECTS 评分与化验单            │  │
│  ├─────────────────────────────────────────────────────────────────────┤  │
│  │ 双轴矩阵多智能体协同网络 (LangGraph): Proposer-Critic-Integrator    │  │
│  └─────────────────────────────────────────────────────────────────────┘  │
└───────────────────────────────────────────────────────────────────────────┘
```

### （一） 前端交互层设计 (Frontend)

前端基于 Vue 3 框架构建，采用组件化开发模式。

- **核心功能**：实现智能问诊、患者管理、急诊倒计时与医学知识学习的统一展示。支持模块快速切换、医学文档（PDF）在线预览、语音录入、以及医学影像/检验报告图片的上传（支持拖拽与多图预览）。
- **流式与动态渲染**：重点实现了基于 `ReadableStream` 的流式数据接收机制，持续接收后端推送的分析结果并实时打字机渲染。同时，前端挂载基于 WebSocket 的实时动态时效面板，实时更新 DNT 和发病时间窗状态。

### （二） 后端服务层设计 (Backend)

后端基于 Spring Boot / Spring WebFlux 构建，负责高并发环境下的核心业务流转与医院系统对接。

- **安全与控制**：通过 JWT 机制实现医生/医学生身份认证与患者权限隔离，利用分布式锁控制急诊高并发环境下的读写冲突。
- **调度与中转**：采用响应式编程模型，通过 `WebClient` 对底层模型服务进行非阻塞的流式调用，并将底层 Python 服务输出的事件流稳定转发至前端，实现端到端的高效数据链路。

### （三） 模型推理服务层设计 (Model)

基于 Python + FastAPI 构建，是系统智能能力的核心大脑。

- **统一入口加载**：集成大语言模型（Qwen-Max / Qwen-Plus / Qwen-Turbo 三级模型矩阵）、多模态视觉模型（Qwen-VL-Max）、混合检索引擎（ChromaDB + BM25）、重排模型（gte-rerank）及 LangGraph 状态机推理模块。
- **异步流式输出**：通过异步生成器（`Async Generator`）持续输出 `thinking`（思考流）、`chunk`（文本流）、`done`（结束标志）等标准事件流，保持前后端一致的流式通信。

### （四） 三层协同与架构优势

- **数据流向**：用户输入/影像上传 $\rightarrow$ 前端发送请求 $\rightarrow$ 后端处理与转发 $\rightarrow$ 模型服务推理 $\rightarrow$ 流式返回结果 $\rightarrow$ 前端实时渲染。
- **架构优势**：这种设计具备极高的可扩展性（可独立升级或替换 AI 模型）、高并发支持（响应式架构）、以及良好的工程落地性，为从开发环境到真实的院内生产部署（如 HIS/PACS 外挂挂载）铺平了道路。

## 四、 核心技术与模型能力

本项目的核心竞争力在于其面向医疗场景深度定制的智能推理与安全拦截能力。

### （一） 检索增强生成（RAG）机制

系统以医学文献与临床指南作为知识底座，实施证据获取前置。

- **混合检索策略**：采用向量检索（ChromaDB）与 BM25 关键词检索并发，兼顾语义相似性与高频医学术语（如"TOAST分型"、"NIHSS评分"）的精确匹配。
- **指南优先策略**：针对脑卒中场景，优先返回权威指南（如 AHA/ASA 指南、中国脑卒中诊疗指南）与最新文献内容，从源头切断大模型"凭空捏造"的幻觉风险。
- **高级 QA 自建引擎**：系统精读医疗 PDF 并自动批量衍生提炼高质量 `Q:A` 对（附带原文页码标签），大幅提升急诊场景下的检索召回率。

### （二） 证据重排与上下文优化

在初步召回文档后，系统引入重排模型（Reranker）进行深度语境打分与二次筛选。

- **语义重排**：集成 `gte-rerank` 重排模型，对混合检索召回的候选证据进行深度语境相关性打分与二次排序，精准计算指南片段与临床主诉的语义相关性。
- **上下文压缩**：通过限制证据长度并进行结构化聚合，避免冗余信息干扰模型推理效率，防止长上下文导致的"迷失在中间（Lost in the Middle）"现象。
- **精准溯源标注**：对采纳的证据强制进行来源标注（包括文献名称、章节、页码），使输出具备 100% 的溯源验证能力。

### （三） 医学多模态联合推理机制 (Multimodal)

在文本分析基础上，系统深度整合了视觉与语音的多模态能力。

- **ASPECTS 评分辅助提示**：通过多模态大模型（Qwen-VL-Max）对急诊头颅 CT 的高密度影、脑沟变浅等区域进行视觉语义理解，自动关联并辅助计算 ASPECTS 评分。
- **检验检查全景关联**：自动读取并解析患者的血小板计数、凝血功能（INR）等报告单图片，将异常指标转化为结构化文本，强制作为禁忌症审查因子的输入。
- **智能意图分流**：系统根据用户提问中的关键词自动识别影像类型（检验报告 / 药品识别 / 通用影像），加载对应的专科提示词模板，实现精准的多模态分析。

### （四） 医学多智能体矩阵协同推理机制（Multi-Agent System）

为解决传统单模型医疗决策盲点多、风险高的痛点，系统基于 LangGraph 创新设计了"**业务专家轴（纵向） $\times$ 决策行为轴（横向）**"的双轴矩阵多智能体协同架构，模拟三甲医院真实临床的"科室会诊"与"三级把关"流程。

```text
                    【 决策行为轴 (横向 LangGraph 拓扑) 】
                     Proposer             Critic            Integrator
                  (方案生成智能体)     (风险审查智能体)    (整合反思智能体)
                  ┌─────────────────┐ ┌───────────────┐  ┌────────────────┐
全科医生 (GP) ───►│ 初始全科病情评估 │ │ 基础高危筛查  │  │                │
神经专家 (NS) ───►│ TOAST分型/时间窗│ │ 神经禁忌症挖掘│──►│ 最终结构化决策  │
临床药师 (CP) ───►│ 药物联合配伍方案 │ │ 溶栓出血风险  │  │  (合规临床报告) │
                  └─────────────────┘ └───────────────┘  └────────────────┘
                            │                  │                  ▲
                            └──────────────────┴── [校验失败拦截] ─┘
                                                (触发异步自愈反思流)
```

- **纵向（医学专家角色定义）**：
  - **全科医生 Agent**：负责全盘审视整体状态，提取主诉、既往史、基础慢病风险与离线语音查房数据转化。
  - **神经专科医生 Agent**：核心决策大脑，专注于特异性表现、NIHSS 评分、CT/MRI 多模态影像特征（ASPECTS评分）及溶栓/取栓时间窗硬管控。
  - **临床药师 Agent**：侧重用药安全，审查检验报告单（如 INR、血小板），严审绝对/相对禁忌症及配伍风险。
- **横向（Proposer-Critic-Integrator 拓扑）**：
  - **Proposer（生成）**：三大专家并行产出初步诊疗子方案。
  - **Critic（审查）**：独立执行"盲审"，重点识别发病时间窗陷阱、多模态禁忌症硬碰撞、出血风险等 6 大类高风险点，对方案进行推翻或质询。
  - **Integrator（反思）**：主导融合专家方案与审查意见。
  - **动态反思自愈循环**：若发现 Critic 提出的硬性规则（如近期胃出血仍建议溶栓）未解决，Integrator 将触发拦截机制，自动拉回上一层节点重试，实现异步自愈反思流。

### （五） 流式推理与自动评测机制

- **Token 聚合机制**：通过 SSE 协议将 thinking（思考过程）和模型文本逐步传递至前端。同时，通过 Token 聚合机制对高频输出进行合并处理，有效减少前端渲染压力，保证长文本医学报告生成的流畅性。
- **基于 RAGAS 框架的自动化评测**：通过构建标准测试集并运行评测脚本，从"回答相关性、事实一致性、上下文精准度"等维度对系统结果进行量化追踪，实现评测驱动的持续迭代。

### （六） 三级模型矩阵与算力分级策略

系统针对不同推理节点的精度与速度需求，部署了三级模型矩阵，实现算力与成本的精细化管理：

| 模型层级 | 模型实例 | 部署节点 | 设计考量 |
| --- | --- | --- | --- |
| **旗舰级** | Qwen-Max | Proposer（方案生成）、Integrator（综合决策） | 高参数模型确保临床推理深度与多步反思质量 |
| **均衡级** | Qwen-Plus | Critic（风险审查）、上下文摘要 | 兼顾推理精度与响应速度，控制审查环节延迟 |
| **极速级** | Qwen-Turbo | 意图识别、极速通道问答、命名模型 | 极低延迟响应，支撑高并发轻量级任务 |

## 五、 创新点与作品亮点

- 🏆 **(一) 证据驱动的医学推理范式创新**：打破传统大模型"先生成再解释"导致的严重医疗幻觉，将证据获取前置。模型在生成答案前必须依赖外部多模态医学知识，实现了从"模型主导"向"证据主导"的范式跨越。
- 🛡️ **(二) 多智能体协同的安全拦截机制**：首创"生成—审查—反思"双轴矩阵架构，Critic 结合多模态检验单识别绝对禁忌症，扮演严苛的上级把关医师。配合状态机锁定与自愈循环拦截机制，在后台实现"方案提出 $\rightarrow$ 挑刺风险 $\rightarrow$ 拦截修正"的安全闭环。
- ⏱️ **(三) 具有时效动态硬管控的绿道路径设计**：将"时间就是大脑"的临床急救理念工程化，引入动态时间窗倒计时与 DNT 追踪。系统能根据时间滴答（Tick）动态调整 Agent 推理决策的优先级，体现了对急诊脑卒中场景的深度定制。
- ⚖️ **(四) 极速通道与安全通道双路径设计**：日常或时间极度敏感的通用问答通过极速检索响应；面对高风险、多模态的临床急诊病例，系统自动升维至多智能体协同推理模式，完美兼顾临床效率与医疗安全。
- 🔄 **(五) 诊疗推理、患者管理、知识检索与住培考核的四维闭环**：AI 分析依赖知识库提供证据，患者管理模块提供动态个体化输入并输出标准病历，教学模块利用真实病例衍生变体进行双盲考核，完成了从"单次问答工具"到"长期服务与教学平台"的转变。
- 💻 **(六) 工程化与可扩展性的系统设计创新**：提示词配置、报告模板、校验规则、CDA/FHIR 结构化映射及模型调度完全实现配置化与解耦管理。不仅开发部署顺畅，也为跨病种（如心血管、肿瘤）的低成本系统迁移打下了坚实基础。

## 七、 百度千帆大模型平台能力集成方案

针对本项目高并发、低延迟、强逻辑反思以及多模态的需求，百度千帆大模型平台可提供**全栈式的核心能力支撑**。以下从模型底座、算力调度、RAG 组件、多模态网关与安全合规五个维度，阐述当前系统架构与千帆平台的能力映射与集成路径。

### 1. 异构模型底座与专有域微调 (Model Base & SFT)

多智能体矩阵对专科知识深度与反思逻辑要求极高，千帆平台的模型底座能力可为核心推理节点提供强力支撑。

- **高参数旗舰底座**：将 **Qwen-Max** 节点（当前用于 Proposer 与 Integrator）迁移至千帆平台的 **ERNIE 4.0 / 4.0 Turbo** 旗舰模型，利用其行业领先的逻辑推理与多步反思能力，进一步提升医疗风险审查的严苛度与综合决策的可靠性。
- **千帆 SFT（监督微调）平台**：利用千帆提供的 Post-training 工具链，导入系统内置的国内脑卒中权威指南及脱敏临床病例，对大模型进行**专科定向微调**，训练出专属于"神经专科 Agent"和"临床药师 Agent"的专有域模型，提升 TOAST 分型和禁忌症识别的绝对准确率。

**当前架构映射**：

| 当前模型 | 千帆平台对应 | 部署节点 |
| --- | --- | --- |
| Qwen-Max（旗舰级） | ERNIE 4.0 / 4.0 Turbo | Proposer、Integrator |
| Qwen-Plus（均衡级） | ERNIE 3.5 / ERNIE Speed | Critic、上下文摘要 |
| Qwen-Turbo（极速级） | ERNIE Lite / ERNE Speed | 意图识别、极速通道、命名 |

### 2. 算力分级与极致性价比控制 (MaaS 成本优化)

针对系统的"极速通道与安全通道双路径设计"以及 Agent 矩阵的并发调用，千帆平台提供统一的模型路由与弹性计费体系。

- **多尺寸模型混用方案**：
  - **日常问答/极速通道/Proposer（方案生成节点）**：调用千帆平台上的 **ERNIE Speed** 或 **ERNIE Lite**。这些模型推理速度极快，Token 成本极低，适合大规模并行生成初步方案。
  - **风险审查 (Critic) / 综合决策 (Integrator)**：升维调用 **ERNIE 4.0** 旗舰模型，用高参数模型的强逻辑确保医疗安全。千帆统一的 API 路由和计费体系能帮系统大幅降低整体运行成本。
- **弹性扩缩容**：千帆平台的 Serverless 推理架构支持按需弹性扩缩，完美匹配急诊高峰期（如冬季卒中高发季）的突发流量需求，无需预先配置固定算力资源。

### 3. 企业级 RAG 加速与高并发检索组件 (AppBuilder / Reranker)

系统的混合检索在面对全量 PubMed 和海量指南时，工程维护与对齐成本高。千帆平台可提供开箱即用的 RAG 基础设施。

- **千帆 AppBuilder / 知识库组件**：直接调用其内置的**高性能向量检索**与**混合路由策略**，替代当前自建的 ChromaDB + BM25 双路检索方案，省去底层繁琐的索引维护与调优工作，同时获得企业级的可用性保障。
- **千帆 Reranker 模型**：替代当前使用的 `gte-rerank`，调用千帆平台经过大规模中文医疗语料预训练的 Reranker 模型，在中文医学场景下具备更精准的语义相关性计算能力，进一步提升证据重排的准确度。

**当前架构映射**：

| 当前组件 | 千帆平台对应 | 能力提升 |
| --- | --- | --- |
| ChromaDB + BM25（自建混合检索） | 千帆 AppBuilder 知识库 | 免运维、企业级高可用、内置混合路由 |
| gte-rerank（通用语义重排） | 千帆 Reranker | 中文医学语料预训练、更精准的医学语义对齐 |

### 4. 多模态视觉与语音网关能力 (Multimodal API)

当前系统通过 Qwen-VL-Max 实现影像与报告单的视觉理解，千帆平台提供对等的原生多模态能力。

- **文心一格多模态视觉接口**：提供图像语义理解能力，用于解析前端上传的**急诊化验单图片**、**心电图图表**及**医学影像报告/截图**，将其转化为高度结构化的 Prompt 文本或直接输出结构化指标，无缝喂给 LangGraph 拓扑网络。
- **语音识别与合成**：千帆语音 API 可替代当前 ASR 方案，实现床旁语音查房记录的实时转写，将医生口述的患者情况自动转化为结构化文本输入。

**当前架构映射**：

| 当前组件 | 千帆平台对应 | 集成方式 |
| --- | --- | --- |
| Qwen-VL-Max（视觉理解） | 文心一格 Vision API | 替换 `MultiModalConversation.call()` 调用 |
| DashScope ASR（语音转写） | 千帆语音识别 API | 替换语音查房转写链路 |

### 5. 医疗合规与内容安全网关 (Safety Guardrail)

医疗行业对"幻觉"和"错误医嘱"零容忍，千帆平台的安全能力可作为系统的最后一道"硬件防火墙"。

- **千帆大模型安全自护（Guardrails）**：在系统的 Integrator 智能体输出最终结构化临床报告或 EMR 病历前，调用千帆的安全检测 API 作为一个底层的"硬件防火墙"，执行敏感词过滤与合规性检查，防止模型输出任何违反医疗伦理或带有错误偏导向的言论。
- **与现有双重校验的协同**：当前系统已实现"静态禁忌症规则引擎（硬拦截） + 动态 LLM 反思（软审查）"的双重校验机制。千帆 Guardrails 作为**第三层独立安全屏障**，与现有机制形成"规则引擎 → LLM 反思 → 千帆安全网关"的三重纵深防御体系，确保医疗输出的绝对安全。

**三重纵深防御架构**：

```text
Integrator 输出 ──► ① 规则引擎硬拦截（禁忌症关键词匹配）
                        │ 通过
                        ▼
                   ② LLM 反思软审查（深层医学逻辑校验）
                        │ 通过
                        ▼
                   ③ 千帆 Guardrails 安全网关（合规性终审）
                        │ 通过
                        ▼
                   最终临床报告输出
```

## 八、 千帆平台集成实施路径

基于上述能力映射，系统向千帆平台的迁移可按以下三阶段渐进实施，确保业务连续性：

### 阶段一：模型底座无缝替换（低风险）

将当前通过 DashScope OpenAI 兼容接口调用的 Qwen 系列模型，逐步替换为千帆平台的 ERNIE 系列模型。由于系统已采用 LangChain 的 `ChatOpenAI` 抽象层，仅需修改 `base_url` 与 `api_key` 配置即可完成切换，无需改动任何业务逻辑代码。

### 阶段二：RAG 基础设施升级（中风险）

将自建的 ChromaDB + BM25 混合检索方案迁移至千帆 AppBuilder 知识库组件，同时将 `gte-rerank` 替换为千帆 Reranker。此阶段需要适配千帆的知识库 API 接口，但可显著降低运维复杂度并提升检索稳定性。

### 阶段三：安全与多模态能力增强（增量集成）

在现有双重校验机制基础上，接入千帆 Guardrails 安全网关作为第三层防御；将 Qwen-VL-Max 视觉理解能力扩展为千帆多模态 API 双路冗余，提升系统的容错能力与合规保障。
