## 一、现状架构问题分析

经过对现有三层流式架构的代码审查，识别出以下核心技术风险与改进空间。现有架构采用 Python yield 生成器 → asyncio.Queue → FastAPI SSE → Java WebFlux WebClient → 前端 fetch + ReadableStream 的链路，整体可用但在健壮性、可观测性与性能上存在明显短板。

### 1.1 连接稳定性缺陷

> **问题等级: P0 - 严重**
> **影响范围:** 全链路

当前架构在 SSE 连接管理方面存在三个关键缺陷:

- **无心跳保活机制:** Python 端 asyncio.Queue 无定时心跳 yield，Java WebClient 无 keepalive 探针。长推理（>30s）期间代理层或负载均衡器可能因空闲超时断开连接，导致前端收到不完整响应。
- **无断线重连协议:** 前端 fetch 使用手动 ReadableStream 解析而非 EventSource，丧失了浏览器内建的自动重连能力。一旦连接断开，整个回答丢失，用户体验极差。
- **无消息序列号:** SSE 事件不携带 id 字段和序列号，即使实现了重连也无法从断点续传，只能重新生成。

### 1.2 背压与流控缺失

> **问题等级: P1 - 高**
> **影响范围:** Python → Java 链路

模型推理速度（特别是 thinking 阶段大量 token 输出）可能远超前端渲染消费能力:

- **asyncio.Queue 无界队列:** Python 端 Queue 未设置 maxsize，极端情况下内存持续增长。
- **Java WebClient 无消费限速:** WebFlux 的响应式背压机制未被正确利用，flatMap 直接消费上游所有数据。
- **前端 onChunk 同步阻塞:** Vue 的 currentTalkList 每次 chunk 都触发响应式更新，高频 chunk 导致渲染帧丢失。

### 1.3 错误处理薄弱

> **问题等级: P1 - 高**
> **影响范围:** 全链路

- **Python 端:** yield error 类型缺少错误码分类、堆栈信息、是否可重试等元数据。
- **Java 端:** concatWith 中的 persistConversation 异常可能导致 SSE 流异常终止，done 事件未发出。
- **前端端:** reject(new Error(data.message)) 丢失了结构化错误信息，无法区分"模型超时"与"模型拒绝回答"。

### 1.4 可观测性不足

> **问题等级: P2 - 中**
> **影响范围:** 运维监控

- 缺乏端到端链路追踪（traceId / spanId）。
- 无 TTFT (Time To First Token)、TPS (Tokens Per Second) 等关键流式指标埋点。
- 无法区分"模型慢"还是"网络慢"导致的用户体感延迟。

## 二、后端改进方案（Java 层）

### 2.1 SSE 心跳保活与优雅关闭

**目标:** 保证长推理场景下 SSE 连接不被中间件/LB 超时断开。

**改进点:**

1. **Java 端注入心跳 Flux:** 在 streamChat() 返回的 Flux 中 merge 一个定时心跳流，每 15s 发送 SSE comment（: heartbeat）。收到 done 事件后取消心跳。
2. **Python 端空闲心跳:** 在 asyncio.Queue 的消费循环中增加 asyncio.wait_for 超时分支，超过 10s 无新 yield 时主动推送 `{"type": "heartbeat"}` 事件。
3. **优雅关闭序列:** done 事件后，Java 端延迟 500ms 发送 SSE comment ": close"，再关闭流。避免前端在解析最后一个 chunk 时连接就被切断。

**参考实现 — Java 端:**

```java
// QuesController.java - 心跳 merge
Flux<String> heartbeat = Flux.interval(Duration.ofSeconds(15))
    .map(i -> ": heartbeat\n\n");

Flux<String> dataStream = streamingService.streamChat(req);

return Flux.merge(dataStream, heartbeat)
    .takeUntilOther(dataStream.filter(s -> s.contains("done")))
    .concatWith(Mono.just(": close\n\n")
        .delayElement(Duration.ofMillis(500)));
```

### 2.2 断线续传协议

**目标:** 连接断开后客户端可从断点恢复接收，不丢失已生成内容。

**改进点:**

1. **SSE 消息 ID 注入:** 每个 SSE 事件携带单调递增的 id 字段（格式: talkId:seq），Java 端在 parseModelLine 转换时注入。
2. **服务端滑动窗口缓存:** Java 端为每个 talkId 维护最近 N 条事件的 Ring Buffer（建议 N=200，覆盖约 2000 tokens）。
3. **Last-Event-ID 恢复:** 前端重连时携带 Last-Event-ID header，Java 端检查缓存，先回放缺失事件再继续实时推送。
4. **前端改用 EventSource 或兼容库:** 推荐使用 eventsource-parser 替代手动 ReadableStream 解析，获得标准 SSE 解析 + 自动重连。

**协议流程:**

```
Client                    Server
  |--- GET /stream ----------->|
  |<-- id:talk123:1 chunk -----|
  |<-- id:talk123:2 chunk -----|
  |      [连接断开]              |
  |                             |
  |--- GET /stream ----------->|
  |    Last-Event-ID:          |
  |    talk123:2               |
  |<-- id:talk123:3 chunk -----|  (从 seq=3 恢复)
  |<-- id:talk123:4 done  -----|
```

### 2.3 结构化错误码体系

**目标:** 前端能精准区分错误类型并展示对应 UI，运维能快速定位问题层级。

| 错误码 | 含义 | 来源层 | 前端处理建议 |
|--------|------|--------|-------------|
| E1001 | 模型推理超时 | Python Agent | 提示用户简化问题 |
| E1002 | 模型拒绝回答 | Python Agent | 展示安全提示 |
| E1003 | 模型 OOM | Python Agent | 自动重试 (max 1次) |
| E2001 | 上游连接失败 | Java Service | 自动重试 + 排队提示 |
| E2002 | 持久化失败 | Java Service | 静默重试, 不阻塞流 |
| E3001 | SSE 解析异常 | 前端 | 降级为完整响应模式 |

**Python 端错误 yield 格式改进:**

```json
{
    "type": "error",
    "talkId": "123",
    "error": {
        "code": "E1001",
        "message": "推理超时, 已达 120s 上限",
        "retryable": true,
        "detail": "qwen_agent.run_clinical_reasoning timeout",
        "timestamp": "2026-03-17T10:30:00Z"
    }
}
```

## 三、模型端改进方案（Python 层）

### 3.1 有界队列 + 背压信号

**目标:** 防止模型端内存无限增长，在消费端处理不过来时给予推理侧信号。

**改进点:**

1. **设置 Queue maxsize:** asyncio.Queue(maxsize=50)，当队列满时 put 操作自动 await 阻塞，形成天然背压。
2. **超时保护:** await asyncio.wait_for(queue.put(item), timeout=30)，超时则丢弃低优先级事件（如 thinking 中间步骤）并记录日志。
3. **优先级区分:** 使用 asyncio.PriorityQueue，error 和 done 事件优先级最高，确保异常时能快速通知下游。

**参考实现:**

```python
# model/main.py
queue = asyncio.PriorityQueue(maxsize=50)

PRIORITY = {"error": 0, "done": 1, "meta": 2, "result": 3,
            "thinking": 4, "heartbeat": 5}

async def producer(gen):
    for item in gen:
        pri = PRIORITY.get(item["type"], 3)
        try:
            await asyncio.wait_for(
                queue.put((pri, item)), timeout=30
            )
        except asyncio.TimeoutError:
            logger.warning(f"Queue full, dropping: {item['type']}")
```

### 3.2 thinking 流压缩与摘要

**目标:** 降低 thinking 类型事件的传输体积，减少网络带宽和前端渲染压力。

当前 qwen_agent 的 run_clinical_reasoning() 在 thinking 阶段会逐 token yield，产生大量小体积 SSE 事件。改进策略:

1. **thinking token 聚合:** 在 Python 端增加 TokenAggregator 中间层，将连续 thinking token 按 100ms 或 20 tokens（先到先触发）聚合为单条事件再放入 Queue。
2. **可选 thinking 降级:** 在 init 事件中协商 thinking 传输级别: full（全部）、summary（仅关键步骤）、none（不传 thinking），由前端在请求参数中指定。
3. **增量快照:** 每 5s 对 thinking 内容生成一个 checkpoint 快照事件，断线重连时只需回放最近快照 + 后续增量。

**TokenAggregator 实现:**

```python
class TokenAggregator:
    def __init__(self, max_tokens=20, max_wait_ms=100):
        self.buffer = []
        self.max_tokens = max_tokens
        self.max_wait = max_wait_ms / 1000
        self.last_flush = time.monotonic()

    def should_flush(self):
        return (len(self.buffer) >= self.max_tokens or
                time.monotonic() - self.last_flush >= self.max_wait)

    def add(self, token):
        self.buffer.append(token)
        if self.should_flush():
            merged = ''.join(self.buffer)
            self.buffer.clear()
            self.last_flush = time.monotonic()
            return merged
        return None
```

### 3.3 端到端链路追踪

**目标:** 每一轮对话可追踪从前端发起到模型返回 done 的完整链路耗时与状态。

1. **traceId 贯穿三层:** 前端在请求 header 中生成 X-Trace-Id (UUID v4)，Java 层透传至 Python 端，所有 yield 事件携带该 traceId。
2. **关键时间戳埋点:** 在 init 事件中增加 t_request_received, t_model_start；在第一个 result chunk 中增加 t_first_token；在 done 事件中增加 t_model_done, t_total_tokens。
3. **指标上报:** Java 端在 done 事件处理时计算 TTFT, TPS, 总耗时，异步写入监控系统。

**改进后的 SSE 事件协议:**

```json

  "newTalk": true,

    "request_received": 1710648000000,
    "model_start": 1710648000150
  }
}

// done 事件 (新增字段)
{
  "type": "done",
  "talkId": "123",

  "stats": {
    "ttft_ms": 320,
    "total_tokens": 1547,

    "result_tokens": 657,

  }
}
```

## 四、Java 侧关键重构建议

### 4.1 持久化解耦

当前 concatWith(Mono.fromCallable(...)) 在流末尾同步执行 persistConversation 和 Redis 写入，存在两个风险: 持久化失败会导致 SSE 流异常终止（done 可能已发但连接异常关闭）；持久化耗时会延迟连接释放。

**改进方案:**

1. **异步事件驱动:** 将 persistConversation 改为监听 done 事件后异步执行，使用 Sinks.one() 触发，不阻塞 SSE 流关闭。
2. **补偿机制:** 持久化失败时写入本地重试队列（如 Redis Stream），定时任务补偿重试，确保最终一致性。
3. **连接释放前置:** done 事件发出后立即允许连接关闭，持久化在后台完成。

```java
// AIStreamingServiceImpl.java - 解耦持久化
return dataFlux
  .doOnNext(event -> {
    if (event.type().equals("done")) {
      // 异步触发, 不阻塞流
      Mono.fromRunnable(() -> {
        try {
          persistConversation(...);
          updateRedisCache(...);
        } catch (Exception e) {
          retryQueue.offer(retryPayload);
          log.warn("Persist deferred to retry", e);
        }
      }).subscribeOn(Schedulers.boundedElastic())
       .subscribe();
    }
  })
  .map(this::toSSEString);
```

### 4.2 parseModelLine 健壮性增强

现有 parseModelLine() 对 Python 端 yield 的 JSON 做逐行 String 解析再 flatMap 重建，存在畸形 JSON 导致整个流终止的风险。

1. **JSON 解析容错:** 单条解析失败时记录日志并跳过，不中断流。使用 onErrorContinue 替代默认的 onErrorStop 语义。
2. **未知 type 兜底:** 收到未知 type（如将来新增的 citation / tool_call）时透传为 meta 事件，不丢弃不报错。
3. **超长行保护:** 单行 JSON 超过 1MB 时截断并发送 warning 事件，防止内存溢出。
