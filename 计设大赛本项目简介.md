# 🧠 多智能体深度检索脑卒中临床辅助决策支持系统 (CDSS)

## 一、项目定位与总体简介

本项目是一个面向**脑卒中（Stroke）临床急救与连续管理场景**的智能医疗辅助决策支持系统（CDSS），旨在通过人工智能技术提升临床辅助决策支持能力。系统以大语言模型为核心，深度融合**检索增强生成（RAG）、双轴矩阵多智能体推理（Multi-Agent System）与多模态医学知识库**，实现了从患者急诊接入、症状/影像输入到辅助分析、医嘱推荐及长期随访的完整闭环。

与传统通用问答系统不同，本项目并非简单依赖模型参数记忆生成结果，而是以权威医学文献与临床指南为知识底座，通过 **LangGraph 状态机驱动的六节点流水线**——**意图识别 → 病例结构化分析 → 证据混合检索 → 多专家并行推理 → 双重校验反思 → 结构化报告生成**——使每一次回答都具备明确证据来源与逻辑依据。系统强调"**证据先行、过程可解释、结果可验证、时效硬管控**"，在保证智能化水平的同时，显著提升了医学场景下的可靠性与安全性。

### 核心设计理念

| 设计原则 | 具体体现 |
|:---|:---|
| **证据先行** | 所有诊疗建议必须基于内置 12 篇权威指南与实时 PubMed 文献检索 |
| **过程透明** | SSE 流式输出完整 Thinking Chain，前端实时渲染推理步骤 |
| **结果合规** | 双重纵深防御：规则引擎硬拦截 + LLM 反思软审查 + 安全卫栏 |
| **时效管控** | 急诊绿道模式自动追踪发病时间窗与 DNT 计时器 |

### 目标用户群体

- 👨‍⚕️ **临床医生与急诊团队**：辅助医生快速查阅脑卒中相关指南、分析急诊病例，在"黄金1小时"内动态追踪发病时间窗与 DNT（入院到溶栓时间），提供精确的诊疗与用药建议。
- 🎓 **医学教学与医学生**：基于真实脱敏病例自动衍生具有"考核陷阱"的数字病例变体，帮助住培医生理解疾病机制与真实临床鉴别诊疗思路。
- 🩺 **患者持续健康管理**：支持个体化健康数据的动态整合与纵向分析，使 AI 能力从单次问答扩展到连续的二级预防与医疗随访过程。

本项目不仅是一个 AI 模型应用，更是一个融合前端交互、后端反应式流处理与分布式模型推理服务的完整系统，具备高并发、高扩展性与实际应用潜力，为构建真实的院内 CDSS 提供了可行的技术路径。

## 二、系统整体功能简介

本项目围绕真实医疗使用场景，构建了四大核心功能模块，形成了从数据输入、知识调用到智能分析与长期管理的完整闭环。

### （一）智能问诊与 AI 辅助分析模块（含急诊绿道）

该模块是系统的核心功能入口，主要通过 **SSE（Server-Sent Events）长连接** 实现流式交互：

- **结构化医学推理**：系统在接收到输入后，首先通过 `IntentNode` 进行意图分类（`consultation` / `knowledge` / `irrelevant`），再通过 `AnalysisNode` 提取结构化临床信息（人口学背景、时间轴、神经系统查体、生命体征、影像学闭环、实验室指标），最终结合多智能体推理生成结构化结果。
- **全维临床视角**：输出维度包括"最可能诊断及依据、解剖定位分析、病理机制解释、置信度评估以及需排除的重要诊断"等，高度贴近真实临床思维。系统强制执行**三步逻辑主线**：解剖定位（Localization）→ 病因定性（Qualitative classification）→ 病理机制整合（Mechanism Integration）。
- **时效动态追踪（急诊绿色通道模式）**：当识别到疑似急性缺血性脑卒中（AIS）患者时，系统界面自动启动**发病时间窗倒计时**与 **DNT（入院到溶栓时间）计时器**。结合多智能体根据患者发病时间（如 $<4.5\text{h}$ 溶栓，$<6\text{h}$ 或 $6\sim24\text{h}$ 取栓），实时高亮提示"距溶栓窗口关闭还剩 X 分钟"，确保急救时效。
- **流式交互体验**：系统支持流式输出，用户可以实时看到 AI 的推理思考过程（Thinking）与阶段性结果，显著提升交互体验与信任感。

### （二）患者管理与个体化分析模块（含 EMR 自动生成）

传统 AI 系统多限于单次随机交互，本项目引入了患者电子档案管理机制。

- **长期档案维护**：用于记录和维护患者基本信息、既往病史、用药史及医生备注等关键数据。后端通过 MyBatis-Plus 操作 MySQL 8.0 持久化存储。
- **动态上下文联动**：单次问诊结束后，后台通过 `ConversationSummaryService` 自动评估每轮对话的信息价值并更新至 `all_info` 上下文字段。后续多轮就诊自动结合历史记录进行个体化风险评估，实现真正的连续性健康管理。
- **结构化电子病历（EMR）对接**：智能体结束推理后，自动将问诊和分析结果转化为标准的高质量**结构化出院小结**或**急诊首程病历**，严格支持 **CDA** 或 **FHIR** 国际医疗数据标准，可无缝导出或对接院内 HIS 系统。

### （三）医学多模态知识学习与文献检索模块

该模块主要用于支持医学学习、科研支持、多模态联合推理与模型知识底座更新。

- **本地与在线双擎**：
  - **本地知识库**：内置 **12 篇**中国脑卒中相关权威指南与共识（如《中国急性缺血性卒中诊治指南2023》《中国重症卒中管理指南2024》《脑血管病防治指南（2024年版）》等），提供在线 PDF 预览（基于 `pdfjs-dist`）与结构化浏览。
  - **在线文献拓扑**：集成 `PubMedService`，通过 NCBI E-Utilities API 一键抓取最新外文高水平文献列表，按证据等级排序（Practice Guideline > Meta-Analysis > RCT > Review > Case Reports）。
- **RAG 强力支撑**：
  - **文档解析管线**：PDF 通过分块策略切分为语义完整的片段（chunk），同时保留原文来源与页码元数据。
  - **AI Batch QA 衍生引擎**：系统精读医疗 PDF 并自动批量衍生提炼高质量 `Q:A` 对（附带原文页码标签），大幅提升急诊场景下的 BM25 关键词检索召回率。采用批次合并策略（每 10 个 chunk 为一批）降低 API 调用次数。
  - **双路混合检索**：ChromaDB 向量语义检索（DashScope `text-embedding-v2` 嵌入）+ BM25 关键词精确检索双路并发，再经 `gte-rerank` 重排模型深度打分。
- **离线语音查房记录**：支持医生在床旁用语音口述患者情况，系统自动通过语音转结构化文本（ASR + NLP）技术，将其转化为标准的文本数据输入给多智能体矩阵。

### （四）医学教学与科研考核模块

针对"医学教学"与"住培考核"场景。

- **全自动医学双盲数字病例生成器**：系统可利用医学大模型，基于真实脱敏病例自动衍生出具有"多维迷惑性陷阱"的模拟病例（例如：故意在历史病史中加入隐蔽的溶栓绝对禁忌症、或微调化验单指标、或修改发病时间窗）。
- **智能考评反馈**：系统用于考核住培医生的鉴别诊断能力与急诊响应速度，并给出 Agent 视角的步骤扣分项与详细考评报告。

> 💡 **功能整体协同说明**：四大模块并非独立存在——智能问诊模块负责实时分析与时效决策，患者管理模块提供动态个体化数据并输出标准病历，医学知识模块提供权威多模态证据来源，教学科研模块则作为系统的应用延伸与能力验证。四者共同构成"**数据—知识—推理—决策—教学**"的完整链路。

### （一）前端交互层设计 (Frontend)

前端基于 **Vue 3.5 + Vite 7** 框架构建，采用组件化开发模式。

| 技术栈 | 版本 | 用途 |
|:---|:---:|:---|
| Vue | 3.5.x | 响应式 UI 框架 |
| Vite | 7.x | 构建工具与开发服务器 |
| Pinia | 3.x | 全局状态管理 |
| morphdom | — | 流式 HTML DOM diff 高效合并 |
| marked + DOMPurify | — | Markdown 安全渲染 |
| pdfjs-dist | — | PDF 在线预览 |
| SSE (EventSource) | — | AI 推理流式接收 |

- **核心功能**：实现智能问诊、患者管理、急诊倒计时与医学知识学习的统一展示。支持模块快速切换（`WorkspaceTabs`）、医学文档（PDF）在线预览（`PdfPreviewModal`）、语音录入、以及医学影像/检验报告图片的上传（支持拖拽与多图预览）。
- **流式与动态渲染**：重点实现了基于 `ReadableStream` 的流式数据接收机制，通过 `morphdom` 进行 DOM diff 高效合并，持续接收后端推送的分析结果并实时打字机渲染。同时，前端挂载基于 SSE 的实时动态时效面板，实时更新 DNT 和发病时间窗状态。

### （二）后端服务层设计 (Backend)

后端基于 **Spring Boot 3.3 / Java 21 / Spring WebFlux** 构建，负责高并发环境下的核心业务流转与医院系统对接。

| 技术栈 | 版本 | 用途 |
|:---|:---:|:---|
| Spring Boot | 3.3.13 | 应用框架 |
| JDK | 21 | 运行环境（虚拟线程支持） |
| Spring WebFlux | — | 响应式非阻塞 Web 框架 |
| MyBatis-Plus | 3.5.5 | ORM 持久层 |
| Redisson | — | 分布式锁 + 限流器 |
| HikariCP | — | 数据库连接池 |
| JWT + Spring Security | — | 认证与权限控制 |

- **安全与控制**：通过 JWT 机制（`TokenInterceptor`）实现医生/医学生身份认证与患者权限隔离，利用 `Redisson` 分布式锁与限流器控制急诊高并发环境下的读写冲突。
- **调度与中转**：采用响应式编程模型，通过 `WebClient` 对底层模型服务进行非阻塞的流式调用（`AIStreamingServiceImpl`），并将底层 Python 服务输出的事件流稳定转发至前端，实现端到端的高效数据链路。

### （三）模型推理服务层设计 (Model)

基于 **Python 3.11+ / FastAPI 0.128** 构建，是系统智能能力的核心大脑。

| 技术栈 | 版本 | 用途 |
|:---|:---:|:---|
| FastAPI | 0.128+ | 异步 Web 框架 |
| LangGraph | 0.2.20 | 状态机推理图编排 |
| LangChain | 0.3.x | LLM 调用与工具链 |
| DashScope SDK | — | 阿里云模型调用（Qwen 系列） |
| sse-starlette | — | SSE 流式输出 |
| ChromaDB | — | 向量数据库 |
| RAGAS | — | RAG 自动化评测 |

- **统一入口加载**：集成大语言模型（Qwen-Max / Qwen-Plus / Qwen-Turbo 三级模型矩阵）、多模态视觉模型（Qwen-VL-Max，通过 DashScope `MultiModalConversation` API 调用）、混合检索引擎（ChromaDB + BM25）、重排模型（gte-rerank）及 LangGraph 状态机推理模块。
- **异步流式输出**：通过异步生成器（`Async Generator`）持续输出 `thinking`（思考流）、`chunk`（文本流）、`done`（结束标志）等标准事件流，保持前后端一致的流式通信。

### （四）三层协同与架构优势

- **数据流向**：用户输入/影像上传 → 前端发送请求 → 后端处理与转发 → 模型服务推理 → 流式返回结果 → 前端实时渲染。
- **架构优势**：这种设计具备极高的可扩展性（可独立升级或替换 AI 模型）、高并发支持（响应式架构）、以及良好的工程落地性，为从开发环境到真实的院内生产部署（如 HIS/PACS 外挂挂载）铺平了道路。

## 四、核心技术与模型能力

本项目的核心竞争力在于其面向医疗场景深度定制的智能推理与安全拦截能力。

### （一）LangGraph 六节点状态机推理引擎

系统基于 **LangGraph StateGraph** 构建了完整的六节点临床推理状态机，每个节点职责明确、流转路径可配置：

```text
┌──────────┐    consultation    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  Intent   │ ───────────────► │ Analysis │ ──►│ Retrieve │ ──►│  Reason  │
│ (意图识别) │                   │(病例分析) │    │(证据检索) │    │(多专家推理)│
└──────────┘                   └──────────┘    └──────────┘    └────┬─────┘
    │                                                               │
    │ irrelevant ──► [Reject] END                                   │
    │ knowledge ───► [KnowledgeAnswer] END                          │
    │                                                               ▼
    │                                                         ┌──────────┐
    │                                                         │ Validate │
    │                                                         │(双重校验) │
    │                                                         └────┬─────┘
    │                                                              │
    │                                              ┌───────────────┼───────────────┐
    │                                              │ pass          │ retry         │ fail
    │                                              ▼               ▼               ▼
    │                                        ┌──────────┐    (回到Reason)   ┌──────────┐
    │                                        │  Report  │                   │  Report  │
    │                                        │(报告生成) │                   │(带警告)   │
    │                                        └────┬─────┘                   └────┬─────┘
    │                                             ▼                              ▼
    │                                            END                            END
```

| 节点 | 实现类 | 核心职责 | 使用模型 |
|:---|:---|:---|:---|
| IntentNode | `intent_node.py` | 意图三分类：`consultation` / `knowledge` / `irrelevant` | qwen-turbo |
| AnalysisNode | `analysis_node.py` | 结构化提取人口学、时间轴、查体、影像、实验室指标 | qwen-max |
| RetrieveNode | `retrieve_node.py` | ChromaDB + BM25 双路检索 → gte-rerank 重排 | — |
| ReasonNode | `reason_node.py` | 三专家并行推理 → LLM 综合生成 PROPOSAL + CRITIQUE | qwen-max |
| ValidateNode | `validate_node.py` | 规则引擎硬拦截 + LLM 软审查双重校验 | qwen-plus |
| ReportNode | `report_node.py` | 按 5 种模板生成结构化临床报告 + 安全卫栏 | qwen-max |

**状态数据模型（ClinicalState）**：系统定义了完整的 `ClinicalState` TypedDict，包含 20+ 个状态字段（如 `patient_info`、`analysis_result`、`retrieved_documents`、`expert_opinions`、`proposal`、`critique`、`validation_result`、`report` 等），确保每个节点间的数据传递类型安全且可追溯。

### （二）检索增强生成（RAG）机制

系统以医学文献与临床指南作为知识底座，实施证据获取前置。

- **混合检索策略**：采用向量检索（ChromaDB）与 BM25 关键词检索并发，兼顾语义相似性与高频医学术语（如"TOAST分型"、"NIHSS评分"）的精确匹配。
- **指南优先策略**：针对脑卒中场景，优先返回权威指南（如 AHA/ASA 指南、中国脑卒中诊疗指南）与最新文献内容，从源头切断大模型"凭空捏造"的幻觉风险。
- **高级 QA 自建引擎**：系统精读医疗 PDF 并自动批量衍生提炼高质量 `Q:A` 对（附带原文页码标签），大幅提升急诊场景下的检索召回率。

### （三）证据重排与上下文优化

在初步召回文档后，系统引入重排模型（Reranker）进行深度语境打分与二次筛选。

- **语义重排**：集成 `gte-rerank` 重排模型，对混合检索召回的候选证据进行深度语境相关性打分与二次排序，精准计算指南片段与临床主诉的语义相关性。
- **上下文压缩**：通过限制证据长度并进行结构化聚合，避免冗余信息干扰模型推理效率，防止长上下文导致的"迷失在中间（Lost in the Middle）"现象。
- **精准溯源标注**：对采纳的证据强制进行来源标注（包括文献名称、章节、页码），使输出具备 100% 的溯源验证能力。

### （四）医学多模态联合推理机制 (Multimodal)

在文本分析基础上，系统深度整合了视觉与语音的多模态能力。

- **ASPECTS 评分辅助提示**：通过多模态大模型（Qwen-VL-Max）对急诊头颅 CT 的高密度影、脑沟变浅等区域进行视觉语义理解，自动关联并辅助计算 ASPECTS 评分。
- **检验检查全景关联**：自动读取并解析患者的血小板计数、凝血功能（INR）等报告单图片，将异常指标转化为结构化文本，强制作为禁忌症审查因子的输入。
- **智能意图分流**：系统根据用户提问中的关键词自动识别影像类型（检验报告 / 药品识别 / 通用影像），加载对应的专科提示词模板，实现精准的多模态分析。

### （五）医学多智能体矩阵协同推理机制（Multi-Agent System）

为解决传统单模型医疗决策盲点多、风险高的痛点，系统基于 **LangGraph** 创新设计了"**业务专家轴（纵向） × 决策行为轴（横向）**"的双轴矩阵多智能体协同架构，模拟三甲医院真实临床的"科室会诊"与"三级把关"流程。

```text
                    【 决策行为轴 (横向 LangGraph 拓扑) 】
                     Proposer             Critic            Integrator
                  (方案生成智能体)     (风险审查智能体)    (整合反思智能体)
                  ┌─────────────────┐ ┌───────────────┐  ┌────────────────┐
全科医生 (GP) ───►│ 初始全科病情评估 │ │ 基础高危筛查  │  │                │
神经专家 (NS) ───►│ TOAST分型/时间窗│ │ 神经禁忌症挖掘│──►│ 最终结构化决策  │
临床药师 (CP) ───►│ 药物联合配伍方案 │ │ 溶栓出血风险  │  │  (合规临床报告) │
                  └─────────────────┘ └───────────────┘  └────────────────┘
                            │                  │                  ▲
                            └──────────────────┴── [校验失败拦截] ─┘
                                                (触发异步自愈反思流)
```

- **纵向（医学专家角色定义，由 `expert_config.yaml` 动态配置）**：
  - **全科医生 Agent**：负责全盘审视整体状态，提取主诉、既往史与慢病风险。配置于 `expert_config.yaml` 的 `generalist` 角色。
  - **神经专科医生 Agent**：核心决策大脑，专注于特异性表现、NIHSS 评分、CT/MRI 多模态影像特征（ASPECTS评分）及溶栓/取栓时间窗硬管控。配置于 `specialist` 角色。
  - **临床药师 Agent**：侧重用药安全，审查检验报告单（如 INR、血小板），严审绝对/相对禁忌症及配伍风险。配置于 `pharmacist` 角色。

- **横向（Proposer-Critic-Integrator 拓扑，由 `ReasonNode` 统一编排）**：
  - **Proposer（生成）**：三大专家通过 `asyncio.gather` 并行产出初步诊疗子方案，每个专家独立调用 LLM 获取专科意见。
  - **Critic（审查）**：LLM 综合三专家意见，同时生成 **PROPOSAL（综合提案）** 和 **CRITIQUE（风险批判）**，重点识别发病时间窗陷阱、多模态禁忌症硬碰撞、出血风险等高风险点。
  - **Integrator（反思）**：由 `ValidateNode` 执行双重纵深防御：
    - **第一层：规则引擎硬拦截**（`rules_config.yaml`）——基于确定性规则检查绝对禁忌症（如 INR > 1.7、血小板 < 100×10⁹/L、3 个月内颅内出血史等），一旦命中立即标记为 `hard_violation`，不可绕过。
    - **第二层：LLM 软审查**——通过 qwen-plus 对提案进行语义级风险审查，识别规则引擎无法覆盖的隐含风险。
  - **动态反思自愈循环**：若校验结果为 `retry`（存在可修正问题），`ValidateNode` 将状态回传至 `ReasonNode` 重新推理；若多次重试仍失败（`fail`），则强制输出报告并附带醒目安全警告。

### （六）结构化报告生成与安全卫栏

`ReportNode` 根据 `report_templates.yaml` 配置，支持 **5 种报告模式**（标准会诊报告、急诊绿道报告、知识问答、教学病例分析、通用分析），每种模板包含固定的章节结构与安全卫栏：

- **安全卫栏（Guardrails）**：报告末尾强制附加"⚠️ 安全提示"模块，包含禁忌症检查结果、置信度声明与免责声明。
- **证据溯源块**：每个诊断建议后附检索来源（文献名 + 章节 + 页码），确保 100% 可追溯。

### （七）流式推理与自动评测机制

- **Token 聚合机制**：通过 SSE 协议将 thinking（思考过程）和模型文本逐步传递至前端。同时，通过 Token 聚合机制对高频输出进行合并处理，有效减少前端渲染压力，保证长文本医学报告生成的流畅性。
- **基于 RAGAS 框架的自动化评测**：通过构建标准测试集并运行评测脚本，从"回答相关性、事实一致性、上下文精准度"等维度对系统结果进行量化追踪，实现评测驱动的持续迭代。

### （八）三级模型矩阵与算力分级策略

系统针对不同推理节点的精度与速度需求，部署了三级模型矩阵，实现算力与成本的精细化管理：

| 模型层级 | 模型实例 | 部署节点 | 设计考量 |
| --- | --- | --- | --- |
| **旗舰级** | Qwen-Max | Proposer（方案生成）、Integrator（综合决策） | 高参数模型确保临床推理深度与多步反思质量 |
| **均衡级** | Qwen-Plus | Critic（风险审查）、上下文摘要 | 兼顾推理精度与响应速度，控制审查环节延迟 |
| **极速级** | Qwen-Turbo | 意图识别、极速通道问答、命名模型 | 极低延迟响应，支撑高并发轻量级任务 |

## 五、创新点与作品亮点

- 🏆 **(一) 证据驱动的医学推理范式创新**：打破传统大模型"先生成再解释"导致的严重医疗幻觉，将证据获取前置。模型在生成答案前必须依赖外部多模态医学知识，实现了从"模型主导"向"证据主导"的范式跨越。系统内置 **12 篇**权威指南 + PubMed 实时文献检索 + AI Batch QA 自动衍生引擎，构建了完整的医学知识底座。
- 🛡️ **(二) 多智能体协同的安全拦截机制**：首创"生成—审查—反思"双轴矩阵架构，Critic 结合多模态检验单识别绝对禁忌症，扮演严苛的上级把关医师。配合 **ValidateNode 双重纵深防御**（规则引擎硬拦截 `rules_config.yaml` + LLM 软审查）与 **LangGraph 状态机自愈循环拦截机制**（retry → 回到 ReasonNode 重推理），在后台实现"方案提出 → 挑刺风险 → 拦截修正"的安全闭环。
- ⏱️ **(三) 具有时效动态硬管控的绿道路径设计**：将"时间就是大脑"的临床急救理念工程化，引入动态时间窗倒计时与 DNT 追踪（前端 `WorkspaceTabs` 组件）。系统能根据时间滴答（Tick）动态调整 Agent 推理决策的优先级，体现了对急诊脑卒中场景的深度定制。
- ⚖️ **(四) 极速通道与安全通道双路径设计**：日常或时间极度敏感的通用问答通过极速检索响应（IntentNode → KnowledgeAnswer）；面对高风险、多模态的临床急诊病例，系统自动升维至六节点完整推理流程（Intent → Analysis → Retrieve → Reason → Validate → Report），完美兼顾临床效率与医疗安全。
- 🔄 **(五) LangGraph 状态机驱动的新一代编排架构**：摒弃传统线性 if-else 链式调用，采用 LangGraph StateGraph 实现声明式状态机定义。支持条件路由（意图分流）、并行节点（三专家 asyncio.gather）、循环边（反思自愈），使整个推理流程具备**可视化、可调试、可扩展**的特性。
- 🔄 **(六) 诊疗推理、患者管理、知识检索与住培考核的四维闭环**：AI 分析依赖知识库提供证据，患者管理模块提供动态个体化输入并输出标准病历，教学模块利用真实病例衍生变体进行双盲考核，完成了从"单次问答工具"到"长期服务与教学平台"的转变。
- 💻 **(七) 工程化与可扩展性的系统设计创新**：提示词配置、报告模板、校验规则、CDA/FHIR 结构化映射及模型调度完全实现配置化与解耦管理。不仅开发部署顺畅，也为跨病种（如心血管、肿瘤）的低成本系统迁移打下了坚实基础。

- 🏥 **临床辅助决策支持（CDSS原型基础）**：在脑卒中等对时间敏感性极高的疾病中，帮助急诊和基层医生快速定位时间窗、适应证与核心禁忌症，提高决策效率并大幅降低医疗事故风险。未来可通过微服务化接口，直接作为智能化外挂接入标准 HIS（医院信息系统）或 PACS（影像系统）。

- 🚀 **可扩展性与推广前景**：得益于优秀的模块化架构与"双轴矩阵"设计，所有专家角色、提示词模板、校验规则、报告模板均通过 YAML 配置文件动态管理，系统具备极强的多模态延展性和跨科室推广能力（如心血管、肿瘤等），为智能医疗系统从实验室走向真实院线环境提供了极具参考价值的落地范本。
